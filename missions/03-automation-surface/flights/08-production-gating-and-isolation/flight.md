## Pre-Flight

### Objective
Re-architect the automation surface's gating so the **production (packaged) binary** is a first-class, safely-gated automation target: the Settings `automationEnabled` toggle is the **sole** thing that binds the MCP server (at launch and live), enabling it is a **human-only UI action** with no programmatic side-channel, the `GOLDFINCH_AUTOMATION_ADMIN` admin tier works on the shipped binary, `--automation-dev` is **demoted to a dev-only convenience** (honored only in unpackaged builds), all **dev runs are profile-isolated** from the installed instance, and the server **falls back to a free port** when its preferred port is taken (env-strict override excepted). Rewrite the CLAUDE.md / `docs/mcp-automation.md` security narrative to match. No spec migration, no `:9222` changes, no new MCP tools.

### Open Questions
- [x] **Dev-profile isolation mechanism** ‚Üí RESOLVED (DD1, operator): key off `app.isPackaged` ‚Äî unpackaged runs redirect `userData` to a `-dev` sibling. Automatic, can't be forgotten.
- [x] **Residual role of `--automation-dev`** ‚Üí RESOLVED (DD4, operator): a dev-only convenience (force-bind + implicit auth-enable + dev-seam + auto-mint), honored **only** when `!app.isPackaged`; a **no-op on the packaged binary**.
- [x] **Can enablement be programmatic?** ‚Üí RESOLVED (DD3, operator): **No.** Human-only / UI-only. Remove the key-mint auto-enable side-effect; guard every other programmatic enable path.
- [x] **Port-fallback shape** ‚Üí RESOLVED (DD6, operator, carried from F7 planning): in dev, `GOLDFINCH_MCP_PORT` binds **exactly or fails loudly**; the setting/default path **falls back to a free port**; the actually-bound port is surfaced; the fallback is ephemeral (does not overwrite the persisted preference).
- [x] **`GOLDFINCH_MCP_PORT` scope (is it production?)** ‚Üí RESOLVED (DD6, operator, 2026-06-17): **dev-only** ‚Äî honored only when `!app.isPackaged`; ignored on a packaged build, where the port comes from the Settings `automationPort` setting + free-fallback. (Today `resolvePort` reads it unconditionally; it was only *moot* in production because the server never bound ‚Äî F8 toggle-binds makes that decision live, hence the explicit scoping.)
- [x] **Live flip-OFF semantics** ‚Üí RESOLVED (Architect): flip-OFF calls `stop()` (full teardown ‚Äî closes listener + connections + session transports, idempotent `mcp-server.js:645`); the indicator clears. - [x] **Dev-profile directory name** ‚Üí RESOLVED: `<userData>` + `-dev` suffix in the standard config location (e.g. `~/.config/goldfinch-dev`) ‚Äî clearly separate, survives `npm` reinstalls.

### Design Decisions

**DD1 ‚Äî Dev-profile isolation via `app.isPackaged` (foundation; sequenced FIRST).**
- Choice: when `!app.isPackaged`, call `app.setPath('userData', <default userData> + '-dev')` **before** anything reads `userData` (before `settings.load(app.getPath('userData'))` at `main.js:868`, and before app `ready`). Every dev launch ‚Äî `dev`, `dev:debug`, `dev:automation`, or a bare `electron .` from source ‚Äî uses the isolated profile (e.g. `~/.config/goldfinch-dev`); the installed binary always uses `~/.config/goldfinch`.
- Rationale: the security property is **"can't be forgotten"** ‚Äî no flag/env to set, so a dev run can **never** pollute the installed profile. This is what makes DD2 (toggle-binds) safe: a dev-left `automationEnabled=true` cannot auto-bind the shipped binary because it lives in a different profile. The F7 debrief is explicit that isolation must land **first or with** toggle-binds (recommendation 4).
- Trade-off: the dev instance does not share the operator's jars/cookies/history (acceptable ‚Äî dev is for surface testing; dogfooding against real content uses the installed binary, which DD2 + DD5 now make drivable). One-time operator action: reset the **already-polluted** shared profile (F7 dev runs flipped `automationEnabled=true` + left dev key hashes there) ‚Äî see Prerequisites.
- **Verified feasible (Architect):** userData is read in exactly three places, all *inside* `whenReady` (`settings.load(getPath('userData'))` `main.js:868`; `shields.js:25` via `load()` at `main.js:867`; `jars.js:70` via `load()` at `main.js:869`) ‚Äî none at module scope. `app.setPath('userData', ‚Ä¶)` at the **top of the `whenReady` callback (before line 867)**, or at module scope, redirects all three; `app.isPackaged`/`app.setPath` are legal pre-`ready` on Electron `^42`. `dev:debug` (unpackaged) is isolated automatically by the same `!app.isPackaged` key ‚Äî no separate handling.
- **Owning AC (leg 1, per the F7 `INTERNAL_PAGES` lesson):** a dev launch must write **all three** of `settings.json`, `shields.json`, AND `containers.json` under the `-dev` dir, and the installed dir must be **byte/mtime-unchanged** across a dev session ‚Äî named explicitly so the ordering invariant is nobody-can-orphan-it.

**DD2 ‚Äî The Settings `automationEnabled` toggle is the SOLE bind gate (the boundary move).**
- Choice: bind the MCP server **iff `automationEnabled === true`** ‚Äî at launch (replacing the `isMcpAutomationEnabled` flag check at `main.js:916,920`) **and live** (flip ON ‚Üí bind; flip OFF ‚Üí unbind: stop the HTTP server, drop active sessions, clear the indicator). In a **packaged** build the toggle is the only thing that binds **and** the only thing that auth-enables. `--automation-dev` is no longer required for binding (see DD4).
- **Net-new wiring, NOT pure infra reuse (Architect correction):** the F5 `stop()` (`mcp-server.js:645-671`) does support unbind-to-zero (closes the listener + all connections + session transports, idempotent), but `rebindMcpServer` (`main.js:160-172`) **always re-binds** and **no-ops when `mcpServer===null`** (`main.js:161`). So leg 2a must add: (a) a start path that creates `mcpServer` **from null** on a cold flip-ON (a packaged build starts with the server unbound), and (b) a "stop-and-stay-stopped" path on flip-OFF ‚Äî hooked off the `automationEnabled` write in `internal-settings-set` (`main.js:641`). This is the bulk of the leg's real work; treat `startMcpServerInstance`/`stop()`/`currentAutomationStatus` as the **building blocks**, not a ready-made toggle.
- Rationale: production opt-in via the toggle alone (operator directive: "the production instance should rely on the toggle alone"). The surface remains **loopback-bound + Origin/Host allow-listed + key-gated**, so the only defense-in-depth given up is "a launch flag is also required" ‚Äî acceptable, and compensated by DD1 (isolation) + DD3 (human-only enable) + the mandatory Architect review.
- **The auth gate is the real authority (why the flip-OFF fallback is safe):** even if a live `stop()` proves leaky, `resolveIdentity` returns null ‚Üí 401 (`mcp-server.js:429,475`) whenever `automationEnabled !== true`, regardless of bind state. So the Adaptation fallback ("flip-OFF 401s + full teardown next restart") is genuinely safe, not a hole.
- **Dev-override interaction (load-bearing ‚Äî keeps the dev harness connected):** the live bind/unbind-on-toggle is the **production** behavior. When the **dev-enable override** is active (`!app.isPackaged` + `--automation-dev`, DD3/DD4), the surface stays **bound + auth-enabled regardless of the persisted toggle** ‚Äî flipping `automationEnabled` off in dev does **not** unbind. This is deliberate: it lets headless tests and the DD9 key-gating behavior test keep their MCP connection while the *persisted* toggle is independently off (the toggle then drives only the UI key-gating + the persisted value, not the dev binding). In production (no override) the toggle is the sole driver of bind/unbind and auth.
- Trade-off: the security boundary now rests on the toggle; DD1 and DD3 are its hard companions, not optional polish.
- **Renderer-copy AC (orphaned cross-process point ‚Äî Architect HIGH):** `settings.js:334-335` ("Not running ‚Äî start Goldfinch with `--automation-dev` to bind the surface") and `settings.js:342-344` ("Takes effect when Goldfinch is launched with `--automation-dev`") hard-code the **old** model and become **false** post-flight. Leg 2a must carry an AC to rewrite these to the toggle-binds reality (DD7 covers CLAUDE.md/docs but NOT this in-UI copy).

**DD3 ‚Äî Enablement is human-only / UI-only; NO programmatic enable (operator directive).**
- Choice: `automationEnabled` may be set `true` **only** by a human flipping the toggle in the settings UI (the existing origin-checked `internal-settings-set` IPC from `goldfinch://settings`, `main.js:641`). Concretely:
  - **Remove the key-mint auto-enable side-effect** ‚Äî `enableAndMintJarKey` (`mcp-server.js:734`) currently flips `automationEnabled = true`. After F8, minting a jar key **creates the credential only**; the human flips the toggle separately. (`mintAdminKey` `mcp-server.js:748` already does **not** write the setting ‚Äî confirmed, no change needed.)
  - **No MCP tool writes settings** ‚Äî confirmed (Architect): `mcp-tools.js`/`engine.js` have no `settings.set` access; the auth gate's `getSettings` is read-only (`mcp-server.js:428`). The surface cannot self-escalate. (Leg 3/2b re-confirms.)
- **Full enumeration of `automationEnabled=true` writers (Architect-verified):** (1) `enableAndMintJarKey` `mcp-server.js:734` ‚Äî *removed* here; (2) the settings-UI toggle IPC `internal-settings-set` `main.js:641` ‚Üê `settings.js:379` ‚Äî the *intended* sole **persisted** human writer; (3) the dev auto-mint stdout block `main.js:941` (calls `enableAndMintJarKey`, so it inherits the side-effect) ‚Äî see the dev-override fix below. After F8, the only writer of the **persisted** `automationEnabled=true` is (2).
- **Dev auto-mint auth fix (Architect MEDIUM ‚Äî corrects my earlier wording):** force-binding the *server* (DD4) does **not** satisfy the auth gate ‚Äî `resolveIdentity` still requires `automationEnabled===true` (`mcp-server.js:429`), so a force-bound-but-disabled dev surface would 401 everything. Resolution: `--automation-dev` (unpackaged) installs an **in-memory dev-enable override** that satisfies **both** the bind decision (DD2) **and** the auth gate, **without writing `automationEnabled` to the store**. This keeps the dev profile's persisted setting `false` (no pollution even within the isolated dev profile) and preserves the human-only invariant for the *persisted* value, while headless tests get a usable surface. The auth gate becomes: identity resolves if `automationEnabled===true` (persisted, human) **OR** the dev-override is active (unpackaged + `--automation-dev`). Auto-mint therefore no longer needs (and must not have) the `enableAndMintJarKey` side-effect.
- Rationale: a cold surface (toggle off, packaged) is unreachable, so nothing connected to it can turn it on ‚Äî the only way from off‚Üíon is a human at the keyboard. (An *admin* session could click the toggle via trusted input, but only **after** a human already enabled the surface to let it connect; it cannot bootstrap from cold. Turning itself *off* is harmless.)
- Trade-off: minting a key and enabling the surface become **two deliberate steps** (today minting auto-enables in one). This is the intended hardening. **Flagged for operator review** ‚Äî say so if you want to keep the mint‚Üíenable coupling.

**DD4 ‚Äî `--automation-dev` becomes a dev-only convenience, honored ONLY when `!app.isPackaged`.**
- Choice: in **unpackaged** builds, `--automation-dev` installs the dev-enable override (DD3: force-binds the surface **and** satisfies the auth gate, without writing the setting ‚Äî so headless behavior tests need no UI), and keeps gating the dev-seam IPC (`automation:dev-invoke`, `main.js:895`) + the auto-mint-to-stdout apparatus. In **packaged** builds, `--automation-dev` is a **no-op** ‚Äî the toggle (DD2) is the only gate.
- Rationale: preserves the headless dev/test ergonomics without a UI, while guaranteeing a **shipped binary cannot be flag-forced** into binding ‚Äî only the human toggle can (reinforcing DD3).
- **Gate at the call sites, NOT inside the pure predicates (Architect):** `src/shared/automation-dev.js` is **electron-free** and used in the renderer preload (`chrome-preload.js:79`) where `app` is unavailable; adding `app.isPackaged` inside `isAutomationDevEnabled`/`isMcpAutomationEnabled` would couple it to electron and break its unit tests + the preload. Instead, AND `&& !app.isPackaged` at the three main-process call sites (`main.js:916` bind, `:895` dev-seam, `:213` additionalArguments). The predicates stay pure.
- Trade-off: the dev force-bind path bypasses the human-only rule ‚Äî acceptable because it is dev-only (unpackaged) and an explicit operator CLI action, never present in a release.

**DD5 ‚Äî The admin tier (`GOLDFINCH_AUTOMATION_ADMIN`) is usable on the production binary.**
- Choice: the admin env gate works in **packaged** builds. It is already independent of `--automation-dev` (`mcp-server.js:434`, `automation-auth.js:84`, `mintAdminKey` `mcp-server.js:749`), so the work is to (a) confirm nothing couples admin to the dev flag or to `!app.isPackaged`, (b) ensure the env-gated admin **UI control** appears + functions on a packaged build, and (c) confirm `mintAdminKey` + admin auth resolve under `app.isPackaged`. Admin still requires the toggle ON (human) **and** a minted admin key.
- Rationale: the "single special flag" the operator wants on production is the admin env (mission Context: "the only special flag should be for the admin key, and that should still be doable in the production instance"). Unblocks dogfooding the chrome's own specs + future sandboxed agent-everything deployments against the real binary.
- Trade-off: none material ‚Äî admin stays off-by-default, key-gated, audited, never issued to external consumers (policy unchanged).

**DD6 ‚Äî Launch-time port free-fallback; `GOLDFINCH_MCP_PORT` is a DEV-ONLY override.**
- **Env scope (operator, 2026-06-17):** `GOLDFINCH_MCP_PORT` is honored **only when `!app.isPackaged`** (consistent with the `--automation-dev` demotion ‚Äî DD4). It is the dev/test apparatus's way to pin a predictable address (`dev:automation` + `GOLDFINCH_MCP_PORT={port}`). On a **packaged** build the env is **ignored** ‚Äî production's only port controls are the Settings `automationPort` UI + the free-fallback. This matches the docs' "dev/test env override" framing and the flight thesis that the production surface is the human UI + the single admin env, and it drops the deliberate `<1024` escape hatch from production.
- **Resolution by build:**
  - **Dev (`!app.isPackaged`):** if `GOLDFINCH_MCP_PORT` is set ‚Üí **env-strict**: bind it **exactly** or **fail loudly** on EADDRINUSE (a deliberate test pin must not silently move). If unset ‚Üí persisted `automationPort` > default, with free-fallback on EADDRINUSE.
  - **Production (`app.isPackaged`):** persisted `automationPort` > default `49707`, **always** free-falls-back on EADDRINUSE. No env path.
- **Keep `resolvePort` pure (Architect principle):** today `resolvePort` reads `process.env.GOLDFINCH_MCP_PORT` unconditionally (`mcp-server.js:152`) and is unit-tested electron-free (`test/unit/automation-port.test.js`). Do **not** add `app.isPackaged` inside it. Instead pass the decision in from the call site ‚Äî e.g. `resolvePort(getSettings, { honorEnv: !app.isPackaged })` ‚Äî so the function stays pure/testable and the electron coupling lives in `startMcpServerInstance` (`main.js`). Update `automation-port.test.js` to cover both `honorEnv` modes.
- **Bound-port capture (Architect ‚Äî concrete change):** `mcpServer.port` is a `const` fixed at construction (`mcp-server.js:254`), and `currentAutomationStatus` reads `mcpStatus.port` set pre-`start()` (`main.js:141`). A `start()` that internally retries a different port would leave both **stale** ‚Üí the UI would show the *attempted* port, not the *bound* one. So `start()` must **expose the final bound port** (mutable field / getter; touch the return shapes at `mcp-server.js:222,254,683`), and `startMcpServerInstance` must capture it **post-`start()`**, not pre. The fallback is **ephemeral** ‚Äî it does not overwrite the persisted `automationPort` preference. The actually-bound port is surfaced via `currentAutomationStatus` + the Settings live-address UI (which already renders `status.port` live ‚Äî `settings.js:322,330`, no renderer change beyond DD2's copy fix).
- **Update the EADDRINUSE hint (`mcp-server.js:624`)** ‚Äî "set GOLDFINCH_MCP_PORT to override" is dev-correct but misleading on a packaged build (env ignored); make the message build-aware (prod ‚Üí it auto-fell-back / use the Settings port control).
- Rationale: the F7 HAT surfaced two instances contending for one port (mission Known Issue). Operator decision (F7 planning): "env strict, else auto-fallback"; env scoped to dev (operator, 2026-06-17).
- Trade-off: a TOCTOU window between a free-port probe and the bind ‚Äî mitigated by **retrying the real `listen` on the next free port on EADDRINUSE** rather than trusting `freePortInRange`'s advisory probe (which self-documents its TOCTOU at `mcp-server.js:166-167`); cap the retries and surface a clear error if exhausted. A fixed-port *production* deployment must set `automationPort` (UI / pre-seeded `settings.json`), not an env var ‚Äî accepted (the agent-everything headless-prod case is out of F8 scope).

**DD9 ‚Äî The toggle gates key GENERATION, not revocation (operator, 2026-06-17).**
- Choice: in the Settings Keys subsection, when `automationEnabled` is **off** the **mint** controls are **disabled** ‚Äî every jar's `Generate key` / `Rotate key` button **and** the env-gated admin `Generate/Rotate admin key` button (`settings.js:560-570,597`). **Revoke is NOT gated by the toggle** ‚Äî it stays governed only by `hasKey` (`settings.js:576,598`); we do **not** auto-revoke on toggle-off, so the operator must still revoke deliberately. The mint controls **enable/disable live** as the toggle flips (the key controller subscribes to `onSettingsChanged`, like the enable IIFE at `settings.js:453`).

Revocation is a safety/cleanup action and must always be available.
- **Owning AC (leg 3, per the F7 orphaned-cross-process-AC lesson):** the key controller (currently an IIFE with no `automationEnabled` awareness, `settings.js:479`) must read the persisted setting + subscribe to `onSettingsChanged`; assert mint disabled‚áîtoggle-off and revoke independent-of-toggle. Backed by the new behavior spec below.
- **Spec-drift fix (leg 3):** `settings-automation.md` Step 8 currently asserts the **old** behavior ("the persisted `automationEnabled` is now `true` ‚Ä¶ a side effect of `enableAndMintJarKey`"). DD3 removes that side-effect; update Step 8 to assert minting does **not** enable + the new gating, so the active spec doesn't pin retired behavior.
