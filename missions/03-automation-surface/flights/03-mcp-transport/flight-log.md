### `drive-tools` — landed 2026-06-13

**Built (code + headless unit coverage; live `tools/list`/`tools/call` over the transport deferred to Leg 6):**
- `src/main/automation/mcp-tools.js` (new) — the SDK-free, Electron-free tool-registry adapter. `buildToolRegistry(getEngine)` → `{ listTools(), callTool(name, args) }`. Imports nothing; returns plain MCP-shaped result objects. Registers the **12 drive tools** 1:1 with the engine op names, each `{ name, description, inputSchema, call(engine, args) }` where `call` is a tiny named→positional mapper (`click` → `engine.click(wcId, x, y, { button, clickCount })`; `scroll` → `engine.scroll(wcId, x, y, dx, dy)`; rest are direct). Reaches engine ops **only** via the injected `getEngine()` result — never imports `tabs`/`nav`/`input`/`resolve` or builds deps, so the `resolveContents` internal-session guard / `isSafeTabUrl` stay authoritative (Leg-1 no-bypass invariant). `getEngine` is called **fresh per `callTool`** (matching the engine's per-call deps discipline); the deref is inside the `try` so a null/throwing engine degrades to `isError`, never a null-deref.
- `src/main/automation/mcp-server.js` — imported `CallToolRequestSchema` (alongside `ListToolsRequestSchema`) and `buildToolRegistry`; replaced the `void _getEngine` no-op with a real `getEngine` (falls back to a `() => { throw 'engine unavailable' }` accessor when absent, so `tools/call` degrades to `isError`); built the registry once; replaced the static empty `ListTools` handler with `async () => ({ tools: registry.listTools() })`; added a `CallToolRequestSchema` handler delegating to `registry.callTool(req.params.name, req.params.arguments)`. SDK stays confined to `mcp-server.js` (the adapter is SDK-free).
- `test/unit/automation-mcp-tools.test.js` (new) — 24 tests, fake recording engine: list shape (12 names + required-field schemas + **no `call` fn leak**); each op's named→positional mapping; success serialization (array / number / boolean for close/activate / `{"ok":true}` for void ops); `openTab` null → normal (not `isError`); each throw class → `isError` with message preserved (incl. close/activate surfacing the `resolveContents` trio, navigate/openTab bad-url, pressKey unknown-key); unknown-tool → `isError`; null/throwing engine → `isError`; `getEngine` called once per `callTool`.

**Chosen success-result shape (the consumer contract; Leg 4 documents it for clients):** the op's actual return value is serialized into one text-content block as JSON via `serialize(value) = value === undefined ? '{"ok":true}' : JSON.stringify(value)`. So: `enumerateTabs` → JSON array; `openTab` → the new `wcId` number **or** `null` (operational — a normal result, not `isError`); `closeTab`/`activateTab` → their **boolean** (`true`/`false`, NOT normalized to `{ok:true}`); the genuinely-void ops (`navigate`, `goBack`, `goForward`, `reload`, `click`, `typeText`, `scroll`, `pressKey`) → `{"ok":true}`. `okResult(value)` → `{ content: [{ type:'text', text: serialize(value) }] }`; `errResult(err)` → `{ content: [{ type:'text', text: String(err?.message ?? err) }], isError: true }`.

**Registry extension-point design (for Leg 3 observe-tools):** the registry iterates a single module-scoped `TOOLS` array, currently `[...DRIVE_TOOLS]`. Leg 3 **appends** its 4 observe tool defs (`captureScreenshot`, `captureWindow`, `readDom`, `readAxTree`) — each with its own `{ name, description, inputSchema, call(engine, args) }` and its own result shaping (screenshots → MCP image content; the `debugger-unavailable` refusal → normal result per DD6). Nothing about the drive tools, `buildToolRegistry`, `listTools`, or `callTool` needs to change — `byName`/`listTools`/`callTool` are all derived from `TOOLS`. Observe was deliberately **not** stubbed here.

## Decisions

### Narrower `--automation-dev`-only gate for the MCP server (Leg 1 design)
**Context**: DD4 calls for the MCP server to be "decoupled from `--remote-debugging-port`," but the existing `isAutomationDevEnabled(argv)` returns true for `--remote-debugging-port` OR `--automation-dev`. Reusing it would co-bind the server under `npm run dev:debug`, re-introducing the multi-session-CDP confound DD10 exists to eliminate (surfaced by the Leg-1 design review, high severity).
**Decision**: add a narrower exported predicate `isMcpAutomationEnabled(argv)` (matches only the `--automation-dev` token) to `src/shared/automation-dev.js`; gate the MCP server on it. The legacy `automation:dev-invoke` seam keeps `isAutomationDevEnabled` unchanged.
**Impact**: CDP-decoupling is structural — `dev:debug` does not start the MCP server — so the Flight-3 DD10 DevTools-conflict test (Leg 6) runs confound-free. Two design-review cycles (approve-with-changes → approve-with-changes); both raised issues incorporated (narrower gate; `before-quit`-primary shutdown with the macOS `window-all-closed` asymmetry handled; module-scoped `mcpServer`; Host-port-mismatch documented as a deliberate allow).

### Multi-session MCP transport (Leg 6 verify-integration — defect fix)
**Context**: Live behavior testing (`mcp-drive-end-to-end`) surfaced a real defect. `mcp-server.js` constructed a **single** `StreamableHTTPServerTransport` once and `mcp.connect(transport)`'d it once at startup. In the SDK's stateful mode (1.29.0) that binds **one session for the app's whole lifetime**: after the first client disconnects, a fresh `initialize` is refused with `-32600 "Server already initialized"` and the old session id is unrecoverable (`-32001 "Session not found"`). Net effect — the server was **single-use per launch**: a client could not reconnect, and a second/concurrent client could not connect. This blocked the mission's external-consumer goal and the end-to-end behavior test.
**Decision**: rework to the SDK's standard **stateful multi-session** pattern. A module-local `Map<string,{ server, transport }>` (closure-scoped in `createMcpServer`) holds live sessions. In `onRequest` the SC7 origin guard still runs **first**, unchanged (a denied request gets `403` before any routing). After the guard, `routeRequest`:
- routes a request bearing a known `Mcp-Session-Id` straight to that session's `transport.handleRequest(req, res)` (covers POST, the GET SSE stream, and DELETE teardown);
- for a POST with no session id, buffers + JSON-parses the body and checks `isInitializeRequest(body)` (now imported from `@modelcontextprotocol/sdk/types.js`); on an initialize it builds a **fresh** `StreamableHTTPServerTransport` (`sessionIdGenerator: randomUUID`, `enableJsonResponse: true`, `onsessioninitialized: (sid)=>sessions.set(sid, entry)`), a **fresh** `Server` via a `buildServer()` factory that re-runs the existing tool-registration path (all 16 tools wired over the same injected lazy `getEngine`), `await server.connect(transport)`, sets `transport.onclose = () => sessions.delete(transport.sessionId)`, then `await transport.handleRequest(req, res, body)` (passing the already-parsed body so the SDK does not re-read the stream);
- a stale/unknown `Mcp-Session-Id` → `404` JSON-RPC; a non-initialize POST or any GET/DELETE with no valid session → `400` JSON-RPC (matching the SDK convention).
`stop()` now closes **all** live sessions (snapshot the map, clear it, `transport.close()` each) plus the http server, and stays idempotent; `start()` stays idempotent and loopback-bound (`127.0.0.1`) with the EADDRINUSE clear-error preserved. The SDK stays confined to `mcp-server.js`; the per-session `Server` still reaches the same live engine via the lazy `getEngine`. A `port` option was added (precedence over `GOLDFINCH_MCP_PORT`/default) so the test can pick a free high port.
**Confirmed SDK 1.29.0 API shapes** (verified against `node_modules`): `StreamableHTTPServerTransportOptions` accepts `sessionIdGenerator?: () => string`, `onsessioninitialized?: (sid)=>void|Promise`, `onsessionclosed?`, `enableJsonResponse?`, `enableDnsRebindingProtection?`; the transport exposes a `sessionId` getter and an `onclose` setter, and `handleRequest(req, res, parsedBody?)` consumes the raw Node pair with an optional pre-parsed body; `isInitializeRequest` is exported from `types.js`.
**Impact**: reconnect and concurrent/multi-client connect now work; the end-to-end behavior test is unblocked. Also closed the flight's open session-model question with the SDK-sanctioned multi-session path rather than the single-session stopgap.

### Regression test — `test/unit/automation-mcp-server.test.js` (headless)
Added a headless unit test (mcp-server.js needs no Electron; a **fake engine** with canned op returns is injected) that drives the real SDK client (`client/index.js` + `client/streamableHttp.js`) against the server on test port **7790**. Four cases: (1) a first client initializes and `tools/list` returns **16** tools; (2) **THE REGRESSION** — after the first client closes, a **second fresh** client initializes + lists successfully (proves reconnect, the thing that was broken); (3) two **concurrent** clients both initialize + list with **distinct** session ids; (4) clean `stop()` then an immediate **restart on the same port** succeeds (no EADDRINUSE) and a fresh client works. Each test stops its server in a `finally` so the port frees.
**Test-harness note (not a server defect)**: each client sets `Connection: close` on its requests. Node's global undici fetch pool would otherwise keep a keep-alive socket and **reuse it across server instances on the same 127.0.0.1 port**; after a `stop()` force-closes sockets, the pooled socket is dead — a later fetch reusing it fails (`fetch failed`) and the dead handle keeps `node --test` from exiting. Per-request connection close means nothing is pooled, so the restart case is deterministic and the process exits with no leaked handles. The real app never hits this (a restart there is a new OS process with a fresh pool). To free sockets promptly on the server side, `stop()` also calls `httpServer.closeAllConnections()` (Node 18.2+) before awaiting `close()`.
