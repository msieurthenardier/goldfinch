# CLAUDE.md

Goldfinch тАФ an Electron desktop browser with a media panel (scan/play/download page media) and a privacy panel (Shields + cookie-jar identities).

## Commands

- `npm start` тАФ run the app
- `npm run dev:automation` тАФ canonical dev launch (`scripts/dev-launch.mjs`, WSL/headless friendly). The launcher passes `--ozone-platform=wayland` when a Wayland socket is reachable (X11 under WSLg swallows the first cross-window click-to-activate; decision logic in `src/main/ozone-platform.js` тАФ it lives in the launcher because Electron resolves ozone before `main.js` runs). A caller-provided `--ozone-platform*` flag wins; non-WSLg X desktops are untouched. `--automation-dev` is the dev-only force-bind for the in-process MCP automation surface (`!app.isPackaged`-gated). See `docs/mcp-automation.md`.
- `npm run dist` тАФ build installers (electron-builder); `npm run pack` for unpacked
- `npm test` тАФ `node --test` over `test/unit/**` (pure security/privacy helpers). For real-environment / UI behavior, drive the running app over the MCP automation surface (`npm run dev:automation`); behavior specs live in `tests/behavior/`.
- `npm run lint` тАФ ESLint, flat config (`eslint.config.mjs`)
- `npm run typecheck` тАФ `tsc --noEmit -p jsconfig.json` (all `// @ts-check` files)
- `npm run a11y` тАФ axe-core audit (`scripts/a11y-audit.mjs`) against the RUNNING app over the MCP surface (zero CDP). Drives five chrome states + eight sheet states; the sheet's wcId is resolved once per run from `enumerateWindows` with **no fallback** (a failed read fails the run loudly). Attach: launch `GOLDFINCH_AUTOMATION_ADMIN=1 GOLDFINCH_AUTOMATION_DEV_MINT=1 npm run dev:automation`, capture the printed key(s), then `export GOLDFINCH_MCP_ADMIN_KEY=<adminKey>` (chrome mode) or `GOLDFINCH_MCP_KEY=<jarKey>` + `--target=<url-substring>` (guest mode). Violations are diffed against the curated `ACCEPTED` allowlist baked into the script тАФ only NEW `(rule id, node-selector)` findings fail. Gate convention: `--tags=wcag2a,wcag2aa,wcag21a,wcag21aa` (axe's full default set adds non-conformance advisories like `region`, the documented app-shell exception); `nested-interactive` is always disabled (the tab/close-button pattern is accepted APG). Default fixture: serve `tests/behavior/fixtures/a11y-media/` via `python3 -m http.server` at `http://127.0.0.1:8000/`. Verify-only тАФ NOT part of headless CI (needs the live GUI). `goldfinch://settings` cannot be audited via `--target` (the eval tools exclude the internal session even for admin); the find overlay is not audited here (its a11y rests on the verbatim attribute carry-over + HAT keyboard pass).

## Architecture

Three processes; understand the boundary before editing.

- **Main** (`src/main/`): `main.js` is the composition root, not the behavior owner. `window-factory.js` constructs each `BaseWindow`/chrome view and owns per-window close teardown; `window-registry.js` owns records `{win, chromeView, tabViews, activeTabWcId}`; `register-{tab,overlay,download,settings,browser}-ipc.js` own their channel families; `app-lifecycle.js` owns ready/restore/activation/quit; `session-runtime.js` owns web-session policy and retention cadence. `shields.js` = Shields config + tracking-param stripping; `jars.js` = container definitions; `trackers.js` = eTLD+1 + tracker classification; `settings-store.js` = app preferences; `internal-ipc.js` = origin-checked bridge for trusted `goldfinch://` pages.
  - **MainтЖФchrome routing has three classes тАФ classify any new send/handler site before writing it**: (1) **sender-resolved inbound** тАФ chrome-sender IPC resolves its window via `registry.getWindowForChrome(event.sender)`, guest-sender via `getWindowForGuest(event.sender.id)`; window lifecycle events (resize/maximize/тАж) route to their OWN chrome via the per-window create closure. (2) **broadcast fan-out** тАФ `broadcastToChromeAndInternal` sends to ALL chromes + internal-session contents once globally. (3) **per-tab owner-routed pushes** тАФ every mainтЖТchrome send tied to a specific tab resolves the tab's owning window's chrome AT EVENT TIME via `getChromeForTab(wcId)` тАФ never fanned out, never left on a focused-window accessor (event-time resolution is what makes move-to-new-window's re-bind automatic). `getChromeContents()` survives only as a last-focused accessor (registry-tracked, seeded at create AND programmatic `win.focus()` тАФ WSLg-safe, membership-validated) тАФ don't build new code against it when a routing class fits.
  - **Lifecycle split**: `window-factory.js` tears down at `close` in this order: find overlay тЖТ tear-off overlay тЖТ menu sheet тЖТ whole-window closed-tab capture/session snapshot тЖТ per-tab side effects + **explicit guest destroy** (Electron never auto-destroys an attached view's webContents). `closed` then removes the record and defers chrome-wc destruction with `setImmediate`. `app-lifecycle.js` owns app-level MCP/store quit ordering. Closing one of N windows never quits; the last close rides `window-all-closed` (non-darwin).
  - **тЪая╕П Never read `win.*` inside `closed`-or-later handlers** тАФ capture teardown inputs (e.g. `const winId = win.id`) at create time. A destroyed-`BaseWindow` property access throws, and an uncaught throw inside the native `closed` emission aborts the listener chain AND permanently wedges the Wayland close path with zero error output.
- **Preloads** (`src/preload/`): `chrome-preload.js` exposes `window.goldfinch.*` (contextBridge) to the chrome. `webview-preload.js` runs in every web page's **main world** (media scanner + fingerprint detect/farble). `internal-preload.js` is a third, distinct preload for trusted `goldfinch://` pages тАФ context-isolated (opposite of `webview-preload.js`), exposes `window.goldfinchInternal` behind an `INTERNAL_ORIGINS` allowlist check (defense-in-depth; main-side `registerInternalHandler` is the authoritative boundary). Keep the three separate.
- **Renderer** (`src/renderer/`): `renderer.js` is the chrome composition/boot root. `chrome/context.js` owns DOM/state references; `tab-controller.js`, `navigation-controller.js`, `media-controller.js`, `privacy-controller.js`, `window-controller.js`, `shortcut-controller.js`, `jars-client.js`, and `overlay-menus.js` own behavior. Each tab is addressed by `wcId`. Popup menus + the new-container dialog render from the main-owned **menu-overlay sheet**, not chrome DOM: chrome controllers own triggers/models/actions; the sheet owns presentation + the APG keyboard contract. The ARIA tab-strip contract remains pinned by `tab-keyboard-operability`.

Key cross-cutting facts:
- **Shields apply to every jar through `session-runtime.js`'s `onSessionCreated`.** Each container/burner/default tab is a session partition; new jars inherit automatically. One `webRequest` listener per event per session тАФ recording and enforcement share it (`applyShields`).
- **Web `WebContentsView`s run `contextIsolation:false`** (preload in the page main world тАФ required for farbling). `nodeIntegration` stays off; preload internals stay module-scoped. Internal views are the opposite (isolated + sandboxed).
- **`asar:false`** in the build config so `src/**` stays as unpacked disk files тАФ required by the internal-page `path.join(__dirname, тАж)` resolver.
- **Find-in-page is a per-window floating overlay `WebContentsView`** тАФ NOT chrome DOM. `src/renderer/find-overlay.{html,js,css}` + its preload (chrome-class trust), managed by the Electron-free `src/main/find-overlay-manager.js`, one lazy instance per window in the registry record's `findOverlay` slot; main owns lifecycle, positioning, focus, and the find session, and fans `found-in-page` counts directly to it. The guest is never inset тАФ the bar floats over full guest bounds. Per-tab `findText`/`findOpen` stay in the chrome renderer, which drives `find-overlay:open`/`close` (per-tab restore sent AFTER `tabSetActive`) and syncs back via `find-overlay-text` (every query) and `find-overlay-closed` (overlay-side user Esc/тЬХ only). Close refocus resolves by SENDER: overlay-side close refocuses the guest; chrome-side moves no OS focus. Hidden on internal tabs and while a sheet menu is open. Not in `tabViews` тЖТ invisible to `enumerateTabs`, admin-tier only.
- Per-tab privacy data: main aggregates тЖТ `privacy-net` IPC тЖТ renderer; preload fingerprint counts тЖТ `sendToHost`. Farble config is fetched synchronously at page load тАФ toggling farble needs a reload.
- Persisted state lives in `userData/app.db` (`documents` table, one row per store тАФ see App database). Legacy JSON files (`shields.json`, `containers.json`, тАж) import once on first boot and are renamed `.migrated`. The jars row is a v2 envelope `{ version: 2, defaultId, containers }`; a v1 bare array migrates on import.
- **Frameless window**: `frame:false` on win/linux (custom minimize/maximize/close controls in the tab strip's `#window-controls` zone, wired via sender-resolved `window-minimize` / `window-toggle-maximize` / `window-close` IPC; Close is the record's `win.close()`, NOT `app.quit()` тАФ the last close rides `window-all-closed`); `titleBarStyle:'hidden'` + traffic lights on macOS. `maximize`/`unmaximize` forwarded per-window over `window-maximized-change` keeps the button's `aria-label`/`data-state` live.
- Drag regions: `#tabstrip` is `drag`; the pill, tabs, and window-control buttons are `no-drag`; a `#tabstrip-drag` spacer guarantees a grab area. `chrome-preload.js` exposes `platform`; the renderer tags `<html>` with `platform-{platform}` for platform CSS.
- **тЪая╕П `WebContentsView` native-surface gotcha тАФ "DOM correct тЙа render correct."** A guest view is an out-of-process native compositing surface, not a DOM box. Bounds/visibility changes around it can mis-render the composited pixels while every `getComputedStyle`/`getBoundingClientRect` read looks correct. A guest bounds change is a discrete `setBounds` STEP, not an animatable quantity тАФ sustaining a chrome-ramps-while-guest-steps mismatch mis-renders on EVERY platform (operator-confirmed on native Windows; not a WSLg quirk). **INVARIANT: never animate chrome layout that resizes/repositions the guest slot; animate only things that float OVER the guest (find bar, menu sheet тАФ view stacking that changes no layout around it), or make the layout change instant.** Rules: (1) prefer compositor-only properties that change no layout around the guest; (2) "DOM geometry reads correct" is necessary-but-insufficient тАФ the acceptance signal is the rendered surface observed by a human in motion (even `captureWindow` misses inter-frame defects; discrete grabs land on settled frames); (3) gate any mechanism touching the guest region on a cheap on-platform spike; (4) don't attribute a render defect to the rig (WSLg) without a cross-platform control.

## Patterns

### `src/shared/` ESM modules

`src/shared/` is real ESM: pure modules use `export`/`import` with explicit `.js` extensions, loaded via `<script type="module">`. Unit tests `require()` the same file (Node тЙе22 synchronous `require(esm)`) тАФ the test runs the exact code the app ships.

- **Two specifier shapes, by consumer.** The `file://` chrome imports disk-relative (`../shared/url-safety.js`). Internal `goldfinch://` pages import **flat** specifiers (`./safe-color.js`) because `internal-page-map.js` owns an exact-match hostтЖТpathnameтЖТfile map, not a directory passthrough тАФ a disk-true relative specifier 404s at boot. Flat imports carry `// @ts-ignore` and type `any` (tsc can't resolve them; backlog, not a bug).
- **CJS-by-design quartet + eslint parse guard.** `automation-dev.js` and `internal-page.js` are **preload-reachable** (renderer-side Node `require` has no `require(esm)`) and MUST stay CJS; `dev-profile.js` and `guest-forward-allowlist.js` stay CJS by zero-benefit ruling. `eslint.config.mjs` binds all four to `sourceType: 'commonjs'`, and the later `src/shared/**` module block's `ignores` entry for them is **LOAD-BEARING** тАФ without it, later-wins re-binds the quartet to `module` and deletes the parse guard (an `export` in a preload-reachable file must FAIL lint, not misbehave at a live boot).
- **Evaluate-seam closed set.** Controller implementations remain module-scoped; `renderer.js`'s composition tail republishes exactly the FD-approved 23-entry set with `Object.assign(globalThis, {тАж})`. Moving an implementation into a controller does not publish it. Growing the set requires an FD ruling.
- **Defer/module pin (permanent house rule, contract-tested).** Every classic `<script>` on a page that also loads a module script must carry `defer` тАФ a non-defer classic executes during parse, before any module. `menu-overlay.html`'s `menu-controller.js` is the product's one remaining classic script (carve-out).
- **New-shared-module checklist**: module script tag only; internal-page consumers need an exact `internal-page-map.js` route; evaluate-reachable entry points go through the closed-set seam (FD ruling); a new contextBridge method still needs its `renderer-globals.d.ts` entry.
- **Two real-boot defect classes `npm test` can't catch**: (1) `mkdirSync`-before-synchronous-persist тАФ `app-db.js`'s `open()` now does it once for the whole substrate; unit fixtures pre-create temp dirs so the suite never surfaces it; (2) classic-script top-level scope collisions тАФ structurally gone for `src/shared/` (module scope); the old collision nets are retargeted as script-tag contract tests (tag-count guard + defer pin + module pin).
- **Grep-AC convention.** Negative/invariant ACs ("this body must not change", "this literal no longer appears") may specify a literal reproducible grep/diff, with each hit judged exempt-or-real, instead of a unit test.
- **MockTimers recipe.** Enable per test (`t.mock.timers.enable(...)` inside the test body тАФ never file-global, which leaks fake time file-wide). Drain with real `setImmediate` around single-step ticks тАФ never one big tick that jumps past intermediate states. Exemplar: `test/unit/automation-find.test.js`.

### Two-point hostile-URL security boundary

`isSafeTabUrl` enforces at two independent points: the renderer `createTab` gate (rejects non-`http:`/`https:`/`about:blank` before `tab-create` fires) and the main `will-navigate` guard (`e.preventDefault()` on the same predicate тАФ blocks bypasses via `window.open` etc.). The shared predicate keeps both in sync.

### Recurring module shapes

Default choices for new logic:
- **Electron-free, injected-deps modules** (main-side): live Electron handles (an fs path, `app`, a view constructor) passed as constructor/call arguments, never `require('electron')` тАФ the logic runs offline under `node --test` with fakes. Exemplars: `automation/engine.js` (`deps()` rebuilt per call), `settings-store.js`, `internal-assets.js`, `menu-overlay-manager.js`, `find-overlay-geometry.js`.
- **`src/shared/` ESM pure decision modules**: e.g. `jar-page-model.js`, `inherit-container.js`. A module consumed only main-side stays plain CJS (`guest-forward-allowlist.js`).

### Internal `goldfinch://` pages тАФ trusted-embedder security model

Three equally-privileged internal origins: `goldfinch://settings`, `goldfinch://downloads`, `goldfinch://jars`. The trust boundary is **"internal vs web," never per-page** тАФ all internal pages share the same `goldfinchInternal` bridge and can call any `registerInternalHandler` channel. Security-critical; treat as such.

- **Scheme + session.** `goldfinch` is registered privileged (`{ standard: true, secure: true }`) at module load (before `app.ready`, which `registerSchemesAsPrivileged` requires); `standard: true` is load-bearing (real origin semantics + CSP). Pages are served from a dedicated in-memory internal session (`session.fromPartition(INTERNAL_PARTITION)`) whose `protocol.handle` is **session-scoped** тАФ the global `protocol` would bind the default session.
- **`INTERNAL_PARTITION` is single-sourced** in `src/shared/internal-page.js` тАФ import byte-for-byte, never re-type. Literal drift silently resolves a different session, the `__goldfinchInternal` marker goes absent, and every gate fails open.
- **CSP is stamped IN the response** (`INTERNAL_CSP` in `main.js`) тАФ custom-protocol responses bypass `webRequest`/`onHeadersReceived`, so that hook would silently never fire. Do not relax the CSP without a security review.
- **The internal route map is owned by `src/main/internal-page-map.js`** (`createInternalPageMap`) and resolved per request by `createResolver` (`internal-assets.js`). It is a host тЖТ pathname тЖТ absolute-file exact map, traversal-proof by design: no path arithmetic on `url.pathname`. New subresources get explicit entries; never a directory passthrough.
- **The four gates** (defense in depth): (1) trust is **call-site provenance** тАФ `createTab(url, container, { trusted: true })`, never inferred from the URL; the page-reachable `onOpenTab` route passes no flag, so web content can never select the internal branch; (2) the trusted branch validates with the `isInternalPageUrl` allowlist; the untrusted branch uses `isSafeTabUrl`, which rejects `goldfinch://`; (3) session-aware `will-navigate` тАФ the internal session may navigate only within `isInternalPageUrl`; web views keep the stricter rule; (4) the protocol handler lives on the internal session alone тАФ web tabs literally have no handler for the scheme.
- **NEVER widen `isSafeTabUrl` to admit `goldfinch://`.**
- The internal view runs `contextIsolation: true`, `sandbox: true`, `preload: internal-preload.js`. The internal session is excluded from the web-content wirings (`applyShields`, downloads) via the `creatingInternalSession` flag + the `__goldfinchInternal` marker.
- **Address bar**: `updateAddressChip(tab)` (called from every address-sync site) sets internal/web/neutral chip state and makes the input `readOnly` on internal tabs. `#address-chip` also triggers the site-info popup (sheet-rendered; model from the shared `deriveSiteInfo`, `src/shared/site-info.js` тАФ the one derivation source). "Site settings тЖТ" (`openSiteSettingsTab`) navigates an existing internal tab тАФ or opens a new trusted one тАФ to `goldfinch://settings/#privacy`. `navigate()` on an internal tab reroutes any non-internal URL to a new web tab (the UX half of isolation; the origin-checked `registerInternalHandler` is the security half).
- **Adding an internal page**: entry tree in `internal-page-map.js`; host in `INTERNAL_HOSTS` (`url-safety.js`); origin in `INTERNAL_ORIGINS` (both `internal-ipc.js` and `internal-preload.js`); open via the trusted `createTab` path тАФ never by relaxing web gates.
- The downloads page's `internal-downloads-*` channels resolve the actionable `savePath` MAIN-SIDE by id. `src/main/jar-ipc.js` is a composition facade: registry/default twins live in `jar-registry-ipc.js`, data/retention twins in `jar-data-ipc.js`, and shared wipe/sweep sequencing in `jar-data-lifecycle.js`. Chrome and internal twins still execute the same named handler bodies. A successful wipe broadcasts `jar-wiped { id }` so chrome controllers close the jar's open tabs.

### `goldfinch://jars` page

Each persistent jar section renders a WAI-ARIA tab strip тАФ History / Cookies / Other site data (`src/shared/jar-panel-model.js`'s `JAR_PANELS`; tablist build in `src/renderer/pages/jars-tabs.js`) тАФ plus a footer (Wipe + Delete, jar-level actions outside every tab). Burner has no tabs (driven by `row.isBurner`, never an id check).

- Tab/tabpanel DOM ids use a **double-hyphen** separator `jar-<jarId>--<panelId>` тАФ `slug()` collapses non-alnum runs to one `-` and never emits `--`, so jar names can't collide with region ids. Hash deep-links (`#jar-<id>--<panel>`) and the scroll-spy key off these ids.
- Every tab renders a live count badge ("<Label> (N)") inside its own label span, seeded by `jars-section-controller.js`'s build-time fetch and kept fresh by panel/invalidation routing тАФ generic section/page render never writes these spans.
- `jars.js` is the jars-page composition root; `jars-page-state.js` owns pure reconciliation, `jars-nav-controller.js` owns navigation/scroll-spy, `jars-section-controller.js` owns persistent/Burner sections and panel composition, and `jars-create-controller.js` owns the stable create panel. Panel modules retain the constructor shape `{ bridge, jarId, mountEl, onError, onActivated? }`.
- **Retention**: `jars-section-controller.js` owns the per-jar header `<select>` and failure revert. `src/main/jars.js` validates/persists `retentionDays`; `jar-data-ipc.js` owns the twin handlers; `jar-data-lifecycle.js` owns immediate sweep sequencing. `session-runtime.js` owns boot/hourly `pruneAllJars`, scheduled by `app-lifecycle.js`.
- History is a fourth data class (`{ id: 'history', storages: null, custom: 'history' }` in `src/shared/jar-data-classes.js`); `handleClearData` dispatches on `d.custom === 'history'` **first**, ahead of the storages-null cache fallback (naive fallthrough would clear the cache while reporting history cleared).
- `handleWipe`/`handleRemove` share `jar-data-lifecycle.js`'s `wipeJarData(ses, jarId)`. `register-browser-ipc.js`'s `identity-new` deliberately does NOT purge history тАФ it is an anti-tracking identity break, not a data-visibility control.

### Settings store (`src/main/settings-store.js`)

The canonical home for app preferences тАФ do not scatter new prefs into ad-hoc constants or `shields.js`.

- **Electron-free**; `userData` injected at `load(userDataPath, opts?)` (called from `init-profile.js`'s `initProfileAndStores`, after `appDb.open`).
- Persists via the app-db `documents` row `'settings'` (transactional UPSERT). Write errors propagate. `load()` **never throws** тАФ each field is validated per-key against `DEFAULTS`; a corrupt field silently repairs to its default, valid siblings kept, unknown keys dropped. The app must boot on a corrupt row.
- `VALIDATORS` maps each settable key to a predicate; `set(key, value)` validates **before** mutating and throws `TypeError` on unknown keys/invalid values. `homePage` requires `isSafeTabUrl(v) && v !== 'about:blank'`.
- **Pluggable `{ serialize, deserialize }` codec seam** тАФ a future `safeStorage`-backed codec replaces only that pair. Do not add encryption now.
- **`toolbarPins` gotchas** (apply to any object-typed key): the no-validator `typeof` fallback wrongly accepts `null`/arrays тАФ object keys need an explicit validator. `NORMALIZERS.toolbarPins` deep-merges the stored value over `DEFAULTS.toolbarPins` at load AND in `set()`, so the object is always fully populated and a new pinnable key needs no migration тАФ it fills with ITS OWN default (media/shields `true`, devtools `false`; there is no general filled-as-`true` rule). `getAll()` deep-copies the nested object so callers can't mutate live config.
- The renderer caches `homePage` (`settingsGet` at startup; the initial `createTab` awaits it) and stays live via the `settings-changed` broadcast on both bridges (`window.goldfinch.onSettingsChanged` / `window.goldfinchInternal.onSettingsChanged`).

### DevTools affordance

Chromium DevTools for the active web tab; human and agent entries share one code path: `window.goldfinch.toggleDevtools(wcId)` / `isDevtoolsOpen(wcId)` (`invoke`, not `send` тАФ the button reflects authoritative state) тЖТ `toggle-devtools`/`is-devtools-open` handlers тЖТ shared `src/main/devtools.js` (`{ mode: 'detach' }` only; docked is backlog). Handlers act on the **passed wcId** (TOCTOU guard тАФ never `activeTab()`) and refuse internal/dead contents. The MCP `openDevTools`/`closeDevTools` ops delegate to the same helper. Triggers: `F12` (before the modifier gate) + `Ctrl+Shift+I`, captured in both the guest `before-input-event` and the chrome keydown fallback (auto-repeat-guarded), plus the pinnable `#toggle-devtools` button (default unpinned) тАФ all three work regardless of pin state, all inert on `goldfinch://`. `devtools-state-changed` (forwarded per-tab from `devtools-opened`/`devtools-closed`) keeps `aria-pressed` live; `wc.isDevToolsOpened()` is the source of truth. Open DevTools takes Chromium's single CDP client slot: `readAxTree`/`scroll` surface the `debugger-unavailable` refusal while the CDP-free ops keep working; closing restores.

### Toolbar pins

- `applyToolbarPins(pins)` toggles `.hidden` per pin тАФ **visibility only**. Tab activation separately toggles `disabled` on internal tabs (buttons act on the active tab's web content). **The pinnable toolbar area is tab-scoped only** тАФ don't put app-scoped controls there (the kebab is the lone exception; future app-scoped controls belong elsewhere).
- Unpinned тЙа disabled: panels and shortcuts (`Ctrl+M`, `Ctrl+Shift+P`) stay active. Focus-restoration guards skip `.focus()` on a hidden button (a silent no-op that strands focus on `<body>`).
- **Right-click Unpin**: `contextmenu` on a pin button (OS menu suppressed) opens the page-context menu in toolbar-mode on the sheet тАФ a single "Unpin {item}" item anchored below the button via `chromePointToSheet`. The dispatch calls `window.goldfinch.unpinToolbarItem(item)` (one-way send, chrome trust domain); main validates `item тИИ ['media','shields','devtools']` and does the read-modify-write тАФ **must spread the full current map** (`{ ...settings.get('toolbarPins'), [item]: false }`; a bare partial would reset the other items to defaults via the normalizer) тАФ then broadcasts `settings-changed`. **Main is the single settings writer**; the renderer never writes. The dispatch also refocuses `els.address` after the send (the anchor button is about to hide). A focused pin button + Context-Menu key double-fires deterministically тАФ the chrome keydown handler early-returns for the three pin buttons; only the `contextmenu` path opens.
- The settings page's Appearance pin toggles read/write the same map over the internal bridge and stay synced via `onSettingsChanged` (handle cleaned up on `pagehide` тАФ see listener-handle pattern below).

### Page context menu

Custom cross-process menu, **web-content-only**. Flow: guest `webContents.on('context-menu')` тАФ wired inside the `!__goldfinchInternal` guard, so internal guests are auto-excluded тАФ `preventDefault()`s the native menu and forwards `{ wcId, params }` to the owning window's chrome (event-time `getChromeForTab`). The chrome builds the model with pure `pageContextModel(params, toolbarItem)` (`src/shared/page-context-model.js`, unit-tested): sections link тЖТ image тЖТ selection тЖТ editable (`editFlags`-gated, omitted-not-disabled) тЖТ spelling тЖТ always Inspect; **NAMESPACED ids** `link:*`/`image:*`/`sel:*`/`edit:*`/`spell:<index>`/`action:*` тАФ spelling dispatches by **INDEX**, so guest strings never round-trip as commands. Opens on the sheet at 1:1 guest coordinates (the sheet covers exactly the guest region тАФ no offset translation); a chrome-focused Shift+F10/Context-Menu key opens an Inspect-only menu via `chromePointToSheet` (the in-guest keyboard case synthesizes a real guest `context-menu` event тАФ same path as a right-click). The dispatch validates every id before acting (vanished params тЖТ no-op, never `createTab(undefined)`; edit actions re-checked against the allowlist; `spell:<i>` bounds-validated) and acts on the wcId **captured at right-click** (TOCTOU). Escape refocus is escape-only: captured `returnFocus` when present, else the address bar тАФ never the guest; no `aria-expanded` for this transient surface. ### Menu-overlay sheet

- **Architecture.** A **per-window lazy singleton** (`src/renderer/menu-overlay.{html,js,css}` + `menu-overlay-preload.js`, chrome-class trust), managed by the Electron-free `src/main/menu-overlay-manager.js` (`createMenuOverlayManager`, injected deps, offline-tested), instantiated per window into the registry record's `sheet` slot тАФ two windows can hold open menus simultaneously. - **Hardening.** All `menu-overlay:*` handlers are **sender-validated by webContents identity** (chrome for 1/2; the sheet's own wc for 4/5 тАФ never payload-declared). ### App database (`src/main/app-db.js`)

Durable substrate for the five small config/state stores тАФ settings, downloads, session, jars, shields тАФ on built-in `node:sqlite`. It is a **clone of `history-store.js`'s seams, not a shared import**: `app.db` is independent of `history.db` (separate corruption/quarantine domains; the write-hot visits table never shares a file with config rows).

- **Substrate ruling**: built-in `node:sqlite` (`DatabaseSync`), no vendored native module тАФ preserves the zero-runtime-dependency identity. It's an experimental API (`ExperimentalWarning` at require time is cosmetic, accepted). **Standing tax: every Electron major bump re-runs the full store suite (`history-store` + `app-db` + all five stores) and treats a `node:sqlite` API break as a first-class migration cost.** Decision records live in the respective `flight.md` files (missions 08/10).
- **One `documents` table, one row per store**: `documents(store TEXT PRIMARY KEY, payload TEXT NOT NULL, updated_at INTEGER NOT NULL)`. `createDocumentStore(name)` returns `{ read(), write(payload, now?), remove() }`; every store write is a transactional whole-document UPSERT of its serialized payload (all five are low-cardinality wholesale-replace workloads тАФ the substrate buys corruption discipline + transactional writes, not indexing). Each store's public API, defaults/validators, legacy-shape handling, and codec seam survive verbatim тАФ codec output lands in `payload`, keeping a future encrypted-codec swap codec-only.
- **WAL family + lifecycle**: `app.db` + `-wal` + `-shm` (`journal_mode=WAL`, `synchronous=NORMAL`). `appDb.open(userDataPath)` runs in `initProfileAndStores`; `app-lifecycle.js` then loads the session store from the open singleton. `app-lifecycle.js` closes app/history DBs at **`will-quit`**, after `before-quit`'s snapshot writers; `close()` checkpoints the WAL and is idempotent.
- **Migration тАФ import once, rename `.migrated`**: at each store's `load()`, row-absent + legacy JSON present тЖТ parse through the store's existing load/repair logic тЖТ write the row тЖТ best-effort rename `<name>.json.migrated` (never fatal). A corrupt legacy file still migrates (the repaired result). Once a row exists, bare legacy files are ignored. **Carve-out**: jars' readable-but-unknown-schema-version `containers.json` never migrates тАФ kept in memory, file untouched, so a future compatible build can recover the original envelope (pinned by `jars-security-forward-version.test.js`).
- **Quarantine тЖТ fresh defaults**: a corrupt `app.db` (+siblings) is quarantined to `.corrupt-<ms-epoch>` and recreated тАФ the app must boot. Stores seed defaults; `.migrated` files are **never re-imported** (arbitrarily stale by then). Jars' post-quarantine reseed is branch-dependent on the legacy-partition probe (intentional parity with pre-migration behavior).
- Shields is at house discipline (injected `userDataPath`, codec seam, transactional writes). Pre-load `save()` stays a silent no-op, but once loaded a row-write failure **propagates** тАФ `shields.set()`/`setPaused()` can reject through their IPC handlers.
- **`user_version` ladder is cumulative STEPS** (v0 тЖТ v1 тЖТ v2 in one open; a fresh profile never pauses mid-ladder). v2 adds `cookie_seen(jar_id, name, domain, path, first_seen_ms)` тАФ retention-sweep bookkeeping, metadata only, via `createCookieSeenStore()` тЖТ `{ insertIfAbsent, deleteByIdentity, deleteByJar, selectExpired }`.

### History store (`src/main/history-store.js`)

Per-jar browsing history on a live `node:sqlite` handle тАФ real schema/indexes/FTS, unlike the document-row stores (needs range queries + full-text search at scale).

- **Recording** (`src/main/history-recorder.js`, `createHistoryRecorder` тАФ Electron-free injected-deps factory, constructed during `app-lifecycle.js` readiness; navigation hooks are installed by `register-tab-ipc.js`). Gates in order: (1) positive registered-jar allowlist; (2) `http:`/`https:` scheme allowlist; (3) per-jar 30 s consecutive-duplicate suppression. A per-`wcId` map backfills late titles; teardown calls `forgetTab(wcId)`.
- **Retention**: `retentionDays` lives on the jar record (integer 1тАУ3650, default 30, validated by `cleanRetention`). `session-runtime.js` owns `pruneAllJars()`; `app-lifecycle.js` invokes it at store open and schedules the hourly `.unref()` cadence. It builds the live `{jarId: days}` map and GCs orphan history rows. Up to ~1 h over-retention accepted.
- **`history-changed { jarId }` invalidation contract**: every mutation broadcasts only `{ jarId }` тАФ never rows or counts; subscribers re-query through their own read path (invalidation-not-snapshot, same as `jars-changed`).
- **IPC twins** (`src/main/history-ipc.js`): each handler body defined once, registered twice тАФ bare `history-*` (chrome trust) + `internal-history-*` (`registerInternalHandler`). Ops validate fail-closed in order (malformed payload тЖТ unknown jar тЖТ bad args); every failure returns a **static, non-interpolated** `history: <op> тАФ <code>` string.
- WAL family (`history.db` + `-wal`/`-shm`), closed at `will-quit`; a corrupt db is quarantined to `.corrupt-<ms-epoch>` and recreated.
- **Two live-probed sqlite gotchas**: never mix a bare `?` with numbered `?1`/`?2` placeholders in one statement (SQLite collapses them onto one bound slot); keep FTS5 on the default `unicode61` tokenizer тАФ a `tokenchars` override turns a whole URL into one token and silently breaks prefix search.

### Jars page data panels

The Cookies and Other-site-data panels share the History panel's constructor shape.

- **Cookies panel** (`jars-cookies-panel.js`): lists the jar's live session cookies via `jars-cookies-list` тАФ **never a `value` field, at any layer** (the handler strips it before it crosses the payload). Per-cookie delete reconstructs the removal URL via `jar-data-helpers.js`'s `cookieUrl` (scheme from `secure`; leading dot unconditionally stripped from `domain` тАФ safe for both host-only and domain-attribute cookies). No confirm step (single cookie = low stakes).
- **Other-site-data panel** (`jars-sitedata-panel.js`): a composite union with **two-tier honest labeling** (Electron's `Session` exposes no origin-enumeration or usage/quota API): "Has stored data" = origins with an on-disk IndexedDB leveldb directory (`originFromIndexedDbDirname`; a literal `_0` port segment is Chromium's default-port sentinel, normalized portless тАФ live-measured, not documented); "Visited тАФ storage unconfirmed" = history origins with no IndexedDB confirmation. **Documented gaps тАФ don't silently paper over**: Local Storage is a consolidated non-origin-keyed store (its origins are unrecoverable); never-visited third-party-only and localStorage-only origins are invisible to both tiers; no usage/quota figure anywhere (raw file sizes would mislabel leveldb overhead as user data). Per-origin delete uses the SAME storage-class set as the retention sweep (cookies excluded); a history row survives an origin clear тАФ the origin downgrades to "visited", it doesn't disappear.
- **Freshness**: panels query on tab-SELECTION (not section visibility) and re-query on their own mutations. No live `cookies.on('changed')` UI subscription (page activity would spam every open jars page). Cross-path staleness is closed by the `jar-data-changed { jarId, classes }` broadcast тАФ fired by `handleClearData` (classes actually cleared), `handleWipe` (unconditional), and the retention sweep's **completion** (never the `setRetention` invoke, which resolves before the async sweep finishes).

### Retention sweep (`src/main/retention-sweep.js`)

Generalizes the per-jar `retentionDays` dial (one window, all classes) from history to cookies + site data, riding the same cadence (boot + hourly, plus an immediate one-jar sweep on a retention edit). The engine is Electron-free/injected-deps тАФ every dependency (`cookieSeen`, `historyOrigins`, `sessionFor`, `cookieUrl`, `now`) is a fake in the unit suite.

- **Two independently-aged classes (deliberate, documented asymmetry)**: cookies age by **first-seen** (Electron exposes no cookie creation time тАФ the reason `cookie_seen` exists); storage ages by **origin last-activity** from history (`expiredOriginsForJar`). Origins with no history signal are never auto-swept (no honest age signal; operator-deletable from the panel).
- **Cookie bookkeeping**: the `session-created` hook attaches a `cookies.on('changed')` listener for jar-matching partitions (partition recovered from `ses.storagePath` via `partitionFromStoragePath` тАФ the hook receives only the Session; never an eager `fromPartition` warm, since `session-created` fires only on a partition's FIRST `fromPartition`). **Overwrite-cause ruling (measured)**: a same-identity value refresh fires an `overwrite`/`removed:true` + `inserted` PAIR тАФ the removal handler SKIPS deletion on `cause === 'overwrite'` (the row keeps its original `first_seen_ms`) and deletes on `explicit`/`expired`/`expired-overwrite`/`evicted`; inserts are `INSERT OR IGNORE`. Cold start: a live cookie with no row gets stamped `first_seen_ms = now` at the next sweep тАФ honest, not backdated.
- **тЪая╕П SEQUENCING invariant (unit-pinned)**: snapshot the aged-out origins BEFORE the same pass's history prune, in BOTH `pruneAllJars` and `handleSetRetention` тАФ the prune deletes exactly the evidence `expiredOriginsForJar` reads; a post-prune read sees nothing for the target case. Order: (1) snapshot, (2) history prune (sync), (3) async cookie/storage sweep from the snapshot.
- Sweeps are async, fire-and-forget, per-jar/per-item isolated; every promise chain ends in a terminal catch (log-and-drop; quit-during-sweep named-accepted). `jar-data-changed` fires on completion with the classes actually swept.
- **Will-quit quiesce guard**: the cookies listener outlives `appDb.close()` тАФ every write checks `appDb.isOpen()` first and is try/caught, so a write-after-close can never throw into Electron's event dispatch and wedge quit.
- **Bookkeeping dies with its data**: per-cookie delete тЖТ delete-by-identity; cookies-class clear / wipe / remove тЖТ delete-by-jar. All fail-soft тАФ a cleanup miss just re-stamps a survivor as "new" (harmless staleness, never a correctness bug).
- Other named-accepted gaps: ~1 h cadence over-retention; a page can refresh a cookie between the sweep's read and its remove (TOCTOU, bounded by the 1-day window floor).

### Address-bar suggestions

Frecency-ranked, prefix-matched history suggestions render on the menu-overlay sheet (the only surface that composites below the toolbar тАФ chrome DOM is occluded by the guest).

- The `suggestions` sheet template is `role="listbox"`/`option` (all text via `textContent`); a row click sends channel-4 `{ id: 'sug:<i>' }` тАФ **INDEX dispatch** (`sanitizeActivatedValue`'s 24-char cap forbids a URL riding `value`).
- **`noFocus` regime**: `deliverInit` is the sheet's sole focus site; a channel-1 `noFocus: true` flag gates that call, so keyboard-driven suggestion repaints never move OS focus off `#address` (existing callers omit it, unaffected). A pointer click still moves native focus тАФ the grace timer exists for exactly that race.
- **Chrome-owned close matrix** (the non-focusing regime means no sheet-side blur/outside-click ever fires): Enter-with-selection or row click тЖТ `activated`; Escape тЖТ `escape` (input keeps focus + text); input emptied тЖТ `input-empty`; `#address` blur тЖТ `blur` via a 150 ms grace timer whose callback re-checks the captured token AND `document.activeElement` (lets a click's activation win the blur-vs-click race); tab switch тЖТ main's tab-switch close, PLUS the brand-new-tab path's explicit `closeSuggestions('navigation')` (`createTab`'s synchronous `activateTab()` precedes any IPC, so the ordinary close never fires there); `activateTab` bumps `suggest.seq` on every activation (invalidates in-flight responses for the previous jar); active-tab navigation тЖТ `navigation`. Main's Ch2 handler validates reasons against `MENU_CLOSE_REASONS` (unknown тЖТ `superseded`).
- **тЪая╕П Ch7-before-Ch6 activated nuance**: the Ch7 sink cancels the timers on every close, but clears `suggest.items`/`selectedIndex` only for reasons OTHER than `activated` тАФ the immediately-following Ch6 `sug:<i>` dispatch must still resolve the clicked row from `suggest.items`; the Ch6 handler finishes the reset. A naive reset-on-every-reason silently breaks every pointer-click navigation.
- **Data path**: `history-store.js`'s `suggest(jarId, query, { limit = 6, now })` тАФ age-bucketed frecency (тЙд4dтЖТ100, тЙд14dтЖТ70, тЙд31dтЖТ50, тЙд90dтЖТ30, else 10; grouped by URL) over the FTS-narrowed subset, via the `history-suggest` twins + chrome-only `historySuggest` bridge. `suggest` has its own 1тАУ10 limit clamp (distinct from the store's 1тАУ500). The 1-char query rides an UNCOVERED prefix path (the FTS `prefix='2 3 4'` index) тАФ a full-token scan, measured separately.
- **Gates**: pure `omnibox-suggest-model.js` `shouldQuery({ focused, isInternal, isBurner, value })` тАФ burner/internal tabs structurally never query (positive allowlist, no negative exclusion). Scoped to the active tab's own jar. Responses re-validate the FULL gate at arrival (`acceptSuggestResponse`) тАФ a stale response never model-replaces a menu the operator didn't ask for.

### Internal-bridge security model (`src/main/internal-ipc.js`)

`registerInternalHandler(ipcMain, channel, handler)` wraps `ipcMain.handle` with a two-condition sender check **before** the handler runs: (1) `INTERNAL_ORIGINS.has(event.senderFrame?.origin)` (a destroyed frame yields `null` тЖТ fails); (2) `event.sender.session.__goldfinchInternal === true` (strict equality). Any mismatch throws `'forbidden: non-internal sender for <channel>'` тЖТ a rejected `invoke` renderer-side. This is the authoritative boundary; the preload's `location.origin` guard is defense-in-depth only.

- **Node-vs-Blink origin gotcha**: Blink serializes a `{standard, secure}` scheme's frame origin to `goldfinch://settings` etc. тАФ the correct values to match. Node's `new URL('goldfinch://settings').origin` returns `'null'`. Do NOT "fix" `INTERNAL_ORIGINS` to match Node's output.
- `internal-*` channels are origin-locked to internal pages; chrome channels (`shields-*`, `settings-get`, `automation:get-activity`, тАж) are bare `ipcMain.handle` on the `file://` chrome trust domain тАФ see the bare-handle rule above; never wrap them with `registerInternalHandler`.

### Overlay-view patterns

Five patterns for any future main-owned overlay view:

1. **Electron's `findNext: true` means "start a NEW find session"**, not "step to next" тАФ the inverse of the `<webview>`-era assumption. Main tracks the last query text and maps UI intent onto engine semantics (same text + Enter = step; changed text = new session). Migration-parity ACs need a spot-check of the underlying API contract, not just byte-parity.
2. **Pending-init queue**: a lazy overlay can receive its first payload before `did-finish-load`; sending immediately drops it. Queue at most one pending init (latest wins), flush on readiness.
3. **Sender-resolved close refocus**: when one IPC channel serves actors with different focus consequences, resolve behavior from the sender's webContents identity тАФ never a payload flag.
4. **Electron-free injected-deps module** for main-side view logic (see Recurring module shapes).
5. **Enumerable vs addressable**: a view kept out of `tabViews` never appears in `enumerateTabs` (document this choice at construction) but stays addressable by wcId тАФ admin-tier only. `enumerateWindows` reports every window's `sheetWcId`/`findWcId` directly (exact O(1) admin read; the old id-space probe walk is retired тАФ `scripts/a11y-audit.mjs` reads `enumerateWindows` with **no fallback**). Absent id = never created (both overlays are lazy); present id + `sheetVisible: false` = instantiated but hidden. `activateTab` on a non-tab wcId returns a harmless `false`.

### Cross-view focus + tab-type idioms

- **Focus-then-send**: any branch forwarding a keyboard-input-expecting action to the chrome view (e.g. the guestтЖТchrome Ctrl+L handoff) MUST call `getChromeContents()?.focus()` **before** `.send(...)` тАФ the send drives only DOM focus; without OS focus you ship a focused-but-untypeable control. (The sheet's own chrome-shortcut branch deliberately omits `.focus()` тАФ not a copyable template.)
- **Decide tab type with `isInternalTab()`/`isWebTab()`, never by reading `.trusted`** тАФ trust is call-site provenance at `createTab`, not a tab-shape flag to re-read.
- **Patch in place, never rebuild, whatever holds `document.activeElement`** on broadcast-rendered pages тАФ a rebuild loses the caret and fires `blur` тЖТ a spurious commit under commit-on-blur models (e.g. jar rename). One rule, all widget kinds тАФ no per-widget carve-outs.

### Tab strip

- **DOM/CSS structure** (`styles.css`): `.tab` is the pure sizing/query container (`flex: 0 1 240px`, `overflow: hidden`, `container-type: inline-size`, **no padding** тАФ its container-query size equals its rect width); `.tab-row` carries the flex layout + padding. Three `@container` disclosure stages: title-hide at 72px, inactive-only close-hide at 56px (the active tab's close is never hidden), padding-compress at 40px on `.tab-row`. `.tab.active` carries a **64px min-width floor** paired with hiding its favicon when narrow тАФ without the floor its close button shrinks to render outside its clipped bounds ("`display:block` in the DOM тЙа painted"); every other tab shrinks floorless. `#tabs` is `overflow: hidden` тАФ no scrollbar at any count, by design.
- **тЪая╕П Container-query self-restyle is a silent no-op**: a `@container` rule whose selector targets the container element itself never applies тАФ no error, no warning, `getComputedStyle` never reflects it. Only descendant selectors work.
- **DOM order is the single order authority.** The renderer's `tabs` Map is idтЖТtab lookup ONLY (iteration order not load-bearing). `orderedTabIds()` is the one order-reading accessor; `commitTabMove(id, targetIndex)` the one order-mutating call. Never `[...tabs.keys()]` for order. `listTabs()`/`enumerateTabs` deliberately stays **creation-order** (FD ruling тАФ consumers address by wcId, never position); `enumerateTabs` is an **all-windows census**: every row carries a registry-stamped `windowId` (the renderer never learns `windowId`), ordered by window creation then per-window creation, returned as a plain array. A mid-boot window contributes zero rows тАФ `enumerateWindows().booted` is the completeness signal.
- **Tab activation under native DnD**: all tab drags (reorder, tear-off, cross-window) are native HTML5 DnD, which fires no trailing `click` тАФ so the tab click handler's activate is **unconditional** (no suppression flag; don't reintroduce one), and a drag activates its tab in `dragstart` (Chrome parity).
- **Keyboard map** (all global chrome shortcuts, never lightbox-deferred, via the one classifier `keydownToAction` in `src/shared/keydown-action.js`): `Ctrl+Tab`/`Ctrl+Shift+Tab` + `Ctrl+PageDown`/`PageUp` cycle visual order with wrap; `Ctrl+1`тАУ`8` jump to position (out-of-range no-op), `Ctrl+9` last; `Ctrl+Shift+T` reopens the last closed tab; `Ctrl+N` new window (lowercase-only; `Ctrl+Shift+N` deliberately unassigned); `Ctrl+Shift+ArrowLeft/Right` reorders the focused tab (strip-scoped, not global). The classifier's digit branch is gated on `!alt` (AltGr layouts report ctrl+alt for digits) and shift-tolerant (AZERTY). **`sheet-accelerator.js` hand-mirrors `keydownToAction`** тАФ every classifier change must land in both files in the same change, or the sheet's accelerator path silently diverges on AltGr locales.
- **Closed-tab stack** (`src/shared/closed-tab-stack.js`): pure, main-owned, bounded (25, oldest evicted) тАФ **one global stack with `windowId`-tagged entries** (documented divergence from Chrome's per-window reopen; entries must outlive their window). Capture at two sites through `closed-tab-capture.js` (shared allowlist body): (1) `tab-close`, before destroy; (2) whole-window capture at the window's `close` (every persist-jar tab in insertion order, `stripIndex` = append sentinel; `win.destroy()` skips `close` so it skips capture тАФ accepted; the block is try/caught so it can never break close). Same positive persist-jar allowlist idiom as the recorder тАФ burner/internal are structurally never captured. Reopen is **renderer-orchestrated**: main pops and returns the entry (pop rule: `stripIndex` honored iff `entry.windowId` === the invoking window's id, else append тАФ whole-window entries therefore always append); the renderer resolves the jar and calls `createTab(url, container, { restoreHistory, insertAt })`. `tab-create` with `restoreHistory` **skips `loadURL` entirely** and calls `navigationHistory.restore({entries, index})` тАФ running both would race. Stack size is PUSH-fed: every mutation broadcasts `closed-tab-stack-changed {size}` to all chromes; the renderer caches via `push-cache.js` (a received push always wins; the boot-seed invoke applies only if no push arrived).
- **Tab context menu**: right-click or Context-Menu key/Shift+F10 on a tab тАФ one `contextmenu` listener per tab button handles both (Chromium dispatches the same native event for both; the document keydown catch-all's exclusion gate covers `target.closest('.tab')` тАФ no parallel keydown listener). Pure model `src/shared/tab-context-model.js`, ids `tab:*`, **omitted-only** (the sheet has no disabled-item shape): `close-others` omitted when sole tab; `close-right` when none to the right; `reopen-closed` at an empty stack; `move-new-window` omitted at last-tab AND for internal tabs (its dispatch sends the strip snapshot to `tab-move-to-new-window`; main re-parents the live guest into a `noBootTab` window). Batch closes are **ordered sweeps**: snapshot the target ids, activate the ANCHOR first when the active tab is a target (Chrome parity), then close each тАФ so `closeTab`'s next-tab fallback never fires mid-sweep. `duplicate` reads live url/title/container renderer-side + the `tab-history-snapshot` invoke (web tabs only), then `createTab(..., { restoreHistory, insertAt: sourceIndex + 1 })`. `reopen-closed` reuses `dispatchChromeAction('reopen-closed-tab')`; `openTabContextMenu` is synchronous (fed by the push cache).

### `action:rowId` confirm-transition key

When a shared confirm area serves N sibling-visible actions on one row (the per-jar data-controls block), key the open/swap transition on the **`(action, rowId)` pair** (compared string-or-null) тАФ never a boolean. A boolean silently breaks the same-row action swap: no rebuild fires, leaving stale copy and handlers wired to the wrong action.

### Automation engine (`src/main/automation/`)

Programmatic drive + observe of tabs. `engine.js` is the single entry point; the pure op modules (`tabs.js`, `nav.js`, `input.js`, `observe.js`, `resolve.js`) hold the logic.

- **Injected deps, Electron-free**: op modules never `require('electron')` at module scope; `engine.js`'s `deps()` is rebuilt per call (a recreated window is always picked up). New ops follow the same shape.
- **`wcId` is the canonical tab handle** тАФ per-tab state (locks, etc.) keys on the stable wcId, never a resolved `wc` reference.
- **Foreground-to-act + re-resolve тАФ with a read/act ASYMMETRY.** Ops that need RENDERED OUTPUT (`actOn`/`actOnPaced`/`scroll`, `captureScreenshot`, `readAxTree`, `printToPDF`, `findInPage`) run resolve тЖТ activate тЖТ **re-resolve** тЖТ act (the pre-activate handle may be stale after the async hop; re-resolving re-applies the exclusions). **`readDom` and `evaluate` do NOT activate** тАФ `executeJavaScript` works on background guests, and a read that steals the operator's foreground is the worse bug. Both halves are pinned in `automation-observe.test.js`; do not "harmonize" them. The raise is at WINDOW scope and lives in `tabs.activateTab` (routes to the owning window's chrome, then `win.focus()` + `registry.noteFocus()` тАФ programmatic focus fires no focus event under WSLg). The predicate is structural: ops that call `activate` raise; ops that don't, don't.
- **`executeJavaScript` is the mainтЖТguest read path тАФ not CDP.** Debugger-free, no single-client conflict, CSP-safe for self-contained reads. CDP `Runtime.evaluate` exists only to bypass page CSP for *library* injection (the a11y audit's case) тАФ the opposite use case.
- **CDP is centralized in `cdp.js` and limited to `readAxTree` + `scroll`**: `withDebuggerSession` + one shared single-client lock Set keyed on wcId + the `debuggerUnavailable` refusal тАФ attach on demand, detach in `finally`, never left attached. A busy debugger returns a discriminated `{ automation: 'debugger-unavailable', тАж }` refusal, not a throw; the shared lock means concurrent `scroll` + `readAxTree` on one wcId can't both attach. Why these two: no pure-JS path to the platform a11y tree, and `sendInputEvent` mouseWheel does NOT scroll `WebContentsView` guests (confirmed live) тАФ `scroll` uses CDP `Input.dispatchMouseEvent` while click/type/key stay on `sendInputEvent`. Route any new debugger need through `cdp.js` тАФ never a second attach path.
- **`automation:dev-invoke` is a deprecated, dev-only seam** (`!app.isPackaged`-gated; new dispatch keys auto-expose). Do not build durable consumers against it тАФ the MCP transport is the supported path.
- **Bind model**: in production the Settings `automationEnabled` toggle is the **sole** bind gate (`shouldBindAutomation` in `src/shared/automation-dev.js`); enablement is human-only (minting a key never enables). In dev, `--automation-dev` force-binds without writing the setting тАФ a complete no-op when packaged (every call site ANDs `!app.isPackaged`); dev runs are profile-isolated (`~/.config/goldfinch-dev`). Headless apparatus: `--automation-dev` + `GOLDFINCH_AUTOMATION_DEV_MINT=1` prints a jar key once (`GOLDFINCH_AUTOMATION_ADMIN=1` also mints the admin key).
- **Management UI** (`goldfinch://settings` тЖТ Automation): enable toggle, live MCP address + `.mcp.json` block, persisted `automationPort` (default 49707) with live rebind. `resolvePort` precedence is dev/prod split: dev honors `GOLDFINCH_MCP_PORT` (env > setting > default; an env pin binds-exactly-or-fails-loudly); production ignores the env (setting > default) with free-fallback to the next free port (the persisted preference is not overwritten; the bound port is surfaced live). Per-jar + env-gated admin key generate/rotate/revoke (show-once plaintext; only hashes stored). An ungraceful client disconnect (dropped standalone GET SSE stream) tears the session down so the activity indicator clears тАФ teardown attaches on the GET stream only; trades away resumable-SSE, correct for these clients.
- **Convention: any handler that mutates settings directly must broadcast `settings-changed` itself** (`broadcastToChromeAndInternal('settings-changed', settings.getAll())`) тАФ only the `internal-settings-set` path does it automatically; otherwise live UI silently lags.

#### `runSerialized` (`src/main/automation/toggle.js`)

The shared mutex serializing `applyEnabledChange`/`rebind` through one `inFlight` promise chain: single in-flight op; rejection-tolerant continuation (a failed prior op doesn't wedge the chain тАФ each op still surfaces its own error to its own caller); identity-guarded `finally` clear (`inFlight === mine`). **Prefer `runSerialized` over a bare `await prior`** for serializing any mutating ops тАФ a bare await wedges on a prior rejection.

#### MCP transport (`@modelcontextprotocol/sdk`)

The sanctioned first runtime dependency тАФ pinned to an **exact** version, imported **only** in `src/main/automation/mcp-server.js` + the `main.js` mount. The engine and op modules stay SDK-free; the MCP layer is a thin adapter.

- Node `http.createServer` bound to **`127.0.0.1` only**, default port 49707, SDK `StreamableHTTPServerTransport` on the raw `(req, res)` pair; stateful sessions (per-connection `randomUUID`).
- **The Origin/Host guard runs FIRST** (`origin-guard.js` тАФ pure, exhaustively unit-tested): a denied request gets a 403 and never reaches the SDK. A loopback bind is necessary but not sufficient тАФ *this browser renders hostile pages*, which can reach a loopback server via DNS rebinding unless Origin/Host are pinned.
- **30 tools**: 18 drive (incl. `dragPointer`, a paced synthetic pointer drag тАФ for NON-tab in-page drags; tab drags are native DnD, which synthetic injection cannot drive; unpaced bursts get coalesced by Chromium, so events dispatch one macrotask apart), 4 observe (`captureWindow` takes optional `windowId` and binds by window identity; `enumerateWindows` тАФ not captures тАФ is the window-topology read), 2 eval (`evaluate`/`injectScript` тАФ `executeJavaScript` in the guest main world, zero CDP; the internal session is excluded even for admin; `evaluate` returns must be JSON-serializable), 2 devtools, 3 admin chrome/app-level (`getChromeTarget` (optional `windowId`), `enumerateWindows` тАФ the single window-topology discovery primitive, one row per window `{windowId, chromeWcId, booted, activeTabWcId, lastFocused, sheetWcId?, sheetVisible, findWcId?, findVisible}`, zero cached state тАФ and `downloadsList`; jar keys get `automation: admin-only`), 1 jar-confined `getHistory` (a jar key reads its own jar; admin reads any known jar with `jarId` required). Result semantics: drive ops тЖТ JSON text; screenshots тЖТ image content; the `openTab`-`null` and `debugger-unavailable` outcomes are **normal results** (read and react), not `isError`. Full consumer reference: `docs/mcp-automation.md`; runnable client: `scripts/mcp-example-client.mjs`. The repo `.mcp.json` ships an empty `mcpServers` map тАФ off-by-default; consumers opt in at their configured port.

## Release / CI

Both workflows are supply-chain hardened; preserve these invariants:

- **All `uses:` pinned to full commit SHAs** with a trailing `# vX.Y.Z` comment тАФ never mutable tags. When bumping an action, **including accepting a Dependabot PR**, resolve the new version's commit SHA and pin to that (Dependabot proposes mutable tags; applying them verbatim regresses the hardening).
- **Least privilege**: top-level `permissions: contents: read`; only the `release`/`update-readme` jobs escalate to `contents: write` per-job.
- `ci.yml` (PRs): `npm ci тЖТ test тЖТ typecheck тЖТ lint тЖТ npm audit --audit-level=high тЖТ package`. A high in a dev-only dep is fixed by bumping the dep, never by lowering the gate.

### Cutting a release (`build.yml`, on `v*` tag push)

1. Bump `package.json` (`npm version patch --no-git-tag-version`) in a `release: vX.Y.Z` commit on `main`.
2. `git tag vX.Y.Z <commit> && git push origin vX.Y.Z`.

Strict semver is enforced (a non-semver `v*` tag refuses to publish); the git tag is the source of truth for the version. A prerelease tag publishes as a GitHub prerelease (doesn't move `latest`, doesn't touch `main`); only a stable tag runs `update-readme` (regenerates the README download links between the `DOWNLOADS` markers, commits as `github-actions[bot]`). Rollback: `gh release delete vX.Y.Z --yes --cleanup-tag`, fix, re-tag. App icon: `build/icon.png`.
