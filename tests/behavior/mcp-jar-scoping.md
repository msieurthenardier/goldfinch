# Behavior Test: MCP surface confines each key to its jar; admin sees all + the chrome composite

**Slug**: `mcp-jar-scoping`
**Status**: active
**Created**: 2026-06-14
**Last Run**: 2026-06-15-13-12-09 (pass — FD-driven; full live matrix via the Flight-5 jars UI: jar-key confinement + out-of-jar/internal/burner refusals + admin-sees-all + admin cross-jar/internal drive. Supersedes the 2026-06-14 partial.)

## Intent

Verify SC8's **confinement half** in the running app: an admitted **per-jar key** can enumerate and drive **only its own jar's** guest tabs, and is refused — with distinct, audit-legible codes — on everything outside that jar: a **cross-jar** target (`automation: out-of-jar`), the **internal** `goldfinch://settings` session (`automation: internal-session`), the whole-window **`captureWindow`** observe (`automation: admin-only`), and **burner** tabs (absent from enumerate, refused on drive). Conversely, an **env-set admin key** (`GOLDFINCH_AUTOMATION_ADMIN` set at launch) sees **at minimum** every persistent jar's guest tabs **plus** the internal settings tab, can drive/observe any of them, and `captureWindow` returns an image.

This needs a behavior test rather than a unit test because the confinement is decided by **session object identity in the live Electron runtime** (`resolveContentsForJar`: `wc.session === fromPartition(jar.partition)`), and the admin relaxation is the live engine built with `{ allowInternal: true }` — these only behave correctly against real interned sessions, real `webContents.id`s, and the real internal-session marker. The unit suite (Leg 2: `scope.js` / `resolve.js` `resolveContentsForJar`) fakes sessions; this spec exercises the confinement end to end with tabs really staged in distinct jars over the loopback transport.

**The scoping authority is the resolved SESSION, not the renderer-reported `jarId`** (the SC8 linchpin, DD7). This is the property under test. See the scope-honesty caveat below: the *spoof* (renderer-`jarId` disagreeing with the session) is **unit-backed**, while the behavior test covers the observable **confinement** that the session-identity rule produces.

**Scope honesty (read before running).**
- **Session-vs-`jarId` authority is unit-backed.** Forcing a renderer-reported `jarId` to disagree with the tab's actual session (to prove the session — not the `jarId` — decides membership) requires **instrumentation** that cannot be staged through the real UI (the UI always creates a tab whose renderer `jarId` matches its partition). So this spec does **not** behavior-test the spoof. The **Leg-2 unit tests** cover it **both directions** (a tab whose `jarId` claims another jar is still scoped by its session; a tab whose `jarId` claims *this* jar but whose session is another jar's is excluded). The behavior test here covers the **observable confinement** — a `personal` key sees only `personal`-session tabs — which is what the session-identity rule produces in practice. Do not over-read this as an end-to-end spoof test.
- **`captureWindow` admin-only on a JAR key is asserted; the admin positive is "at minimum".** The admin enumerate is phrased **positively** ("at minimum all persistent jars + the internal tab") and carries **no negative** — the admin engine runs `allowInternal: true` with **no** jar filter, so it **also** sees burner tabs. Burner exclusion is asserted **only on the jar-key rows**, where it holds.

## Preconditions

- **Apparatus — the enable/mint helper (EXISTS; landed in Flight 4).** Identical to the sibling `mcp-auth-gating` spec: an **env-gated auto-mint-to-stdout in `main.js`** that is **already built** — `shouldAutoMint(argv, env)` gates it on the exact `--automation-dev` token **AND** `GOLDFINCH_AUTOMATION_DEV_MINT=1`.
  - When `shouldAutoMint(argv, env)` is true, the app calls `mintJarKey(jarId, settings, jars)` (and `mintAdminKey(settings)` when `GOLDFINCH_AUTOMATION_ADMIN` is set), which mints the key **hash only**, and **prints `{ key, adminKey }` ONCE to stdout**. The surface is enabled in dev by the **in-memory dev-enable override** (DD3/DD4, Flight 8) — `!app.isPackaged && isMcpAutomationEnabled(argv)` — NOT by the mint (minting no longer flips `automationEnabled`; enabling is human-only via the toggle, or in dev via the override).
  - For this spec the minted jar key must be for the **`personal`** jar (so the staged `personal` tabs are in-jar and the `work`/burner/internal tabs are out-of-jar). `mintJarKey`/`mintAdminKey` live in `src/main/automation/mcp-server.js`, and the boot-time auto-mint-to-stdout wrapper that calls them landed in Flight 4. No new apparatus is needed to run this spec.
  - The Executor **captures the printed `key` (jar runs) and `adminKey` (admin run) from stdout at launch** — the plaintext is shown once.
- **Port (load-bearing for every URL below).** Runs pin the listen port with **`GOLDFINCH_MCP_PORT`** for determinism (new default is `49707`). Every `127.0.0.1:$GOLDFINCH_MCP_PORT` below resolves to the pinned port; export it once at launch and reuse it in all curl/SDK calls.
- **How the key attaches to the client.** The minted plaintext is presented as `Authorization: Bearer <key>` on the **SDK client transport** via `requestInit.headers`:
  ```js
  const port = process.env.GOLDFINCH_MCP_PORT || 49707; // pinned per run; new default 49707
  const transport = new StreamableHTTPClientTransport(
    new URL(`http://127.0.0.1:${port}/mcp`),
    { requestInit: { headers: { Authorization: `Bearer ${key}` } } }
  );
  ```
  (the `connectClient(key)` pattern). The Bearer rides every request the transport sends.
- **Jar-key run (relaunch 1):** Goldfinch launched via `npm run dev:automation` with `GOLDFINCH_AUTOMATION_DEV_MINT=1` and `GOLDFINCH_AUTOMATION_ADMIN` **unset** — mints the `personal` jar key; `adminKey` is `null`. MCP server up on `127.0.0.1:$GOLDFINCH_MCP_PORT/mcp`.
- **Admin run (relaunch 2 — env-gated variant):** Goldfinch relaunched via `npm run dev:automation` with **both** `GOLDFINCH_AUTOMATION_DEV_MINT=1` and `GOLDFINCH_AUTOMATION_ADMIN=1` — additionally mints the admin key (non-null `adminKey`). The env var is read at process start; admin-off (jar-key run) vs admin-on (admin run) are **separate launches**, never a mid-run toggle. **Re-stage the tabs each relaunch** — burners are ephemeral and persistent-jar tab state is per-session; the setup row (Step 1's setup) must be re-performed for the admin run.
- **Real-environment setup (performed via the Goldfinch UI — the jar/container `▾` switcher).** Stage, before the assertion steps:
  - At least one tab in the **`personal`** jar (e.g. `https://example.com` opened while `personal` is the active container).
  - At least one tab in the **`work`** jar (e.g. `https://example.org` opened while `work` is the active container) — the cross-jar target.
  - One **burner** tab (a renderer-only `burner:N` session) — the unautomatable target. Note: burners do not survive a relaunch; re-create per run.
  - The internal **`goldfinch://settings`** tab open (kebab ⋮ → Settings) — the internal-session target.
- **Determining target wcIds.** The Executor needs each tab's `webContents.id`. For the `personal` key these come from its own `enumerateTabs` (only `personal` tabs). For the **out-of-jar** targets (`work` tab, burner, internal) — which a jar key cannot enumerate — the wcIds are obtained from the **admin run's** `enumerateTabs` (which lists all + internal) OR from the launch-side stdout, then used as drive targets in the jar-key run. (The point of the cross-jar drive rows is that a jar key, *given* an out-of-jar wcId, is still refused.)
- `curl`, Bash, and an MCP client (SDK client / adapted `scripts/mcp-example-client.mjs` / Claude Code MCP session) are available.
- **Apparatus disqualification:** the `chrome-devtools` MCP does **NOT** qualify (launches its own browser → false pass). Apparatus = SDK client over `127.0.0.1:$GOLDFINCH_MCP_PORT`, app via `npm run dev:automation`. Not the `:9222` CDP path.

## Observables Required

- mcp (MCP tool results over the loopback transport — `enumerateTabs` arrays, drive-op results, `captureWindow` image blocks, and the `isError` flag + the discriminated `automation: <code>` text on refusals — measured via the MCP client connected with the Bearer header for the bound identity)
- browser (rendered tab state: which jars' tabs exist, the internal settings tab, the staged burner — staged via the UI and cross-referenced against enumerate listings)
- shell (stdout capture of the minted `{ key, adminKey }` at launch — measured via Bash)

## Out of Scope

- **The 401 auth gate / off-by-default / per-key rejection.** Whether a key is *admitted at all* is the SC8 **authentication half**, covered by the sibling `mcp-auth-gating` spec. This spec assumes an admitted identity and tests what that identity may then **see and drive** (confinement). 
- **SC7 origin/host/bind transport guard.** The 403 origin guard (covered by `mcp-loopback-origin-guard`) runs ahead of everything; not re-asserted here.
- **Driving the chrome renderer (`mainWindow.webContents`).** The FD scope decision (Leg 2 flight log) defines admin's "sees all + the chrome" for THIS flight as cross-jar guest visibility + the internal tab + `captureWindow`'s whole-window composite — **NOT** driving the chrome renderer, whose wcId is structurally undiscoverable via `listTabs()` (deferred to Flight 6). The admin assertions above are written to this re-scoped capability; do not read Step 8/9/10 as driving the chrome renderer.
- **Session-vs-`jarId` spoof, end to end.** Not behavior-tested here (needs instrumentation the UI cannot stage). It is **unit-backed** both directions by the Leg-2 `scope.js`/`resolve.js` tests. See the scope-honesty caveat in Intent. The behavior test covers the observable confinement the session-identity rule produces.
- **openTab jar-targeting (implemented — leg 3 / DD3).** A jar key's `openTab` now lands the new tab in its own jar (or is refused if an unknown `jarId` is supplied — `automation: unknown-jar`); attempting to open a tab in a foreign jar returns `automation: out-of-jar`. The old "cannot target the jar" caveat no longer applies. The tab is enumerable and confined to the calling key's jar, so confinement is now enforced at create-time, not just at enumerate/drive time. This behavior is not asserted as a new confinement step here (the jar-key rows above cover enumerate + drive confinement); it is recorded informally so readers of this spec are not misled by the former v1-limitation language.
- **Audit recording of the refusals/successes (DD8).** That each refusal's discriminated code and each success is recorded in the audit ring is a separate concern (the audit-data leg), not asserted here.

## Variants (optional)

- The **admin run is the env-set variant** of the jar-key run (the only launch difference is `GOLDFINCH_AUTOMATION_ADMIN`). It is written as a numbered run because it shares the mint apparatus and re-staging, and is the positive counterpart to the jar-key refusals.
- Could later parametrize the cross-jar drive (Step 3) over additional jar pairs (`banking` ↔ `personal`) and additional drive ops (click/typeText/scroll) once the membership path is the focus — the membership check is op-agnostic (every `WCID_FIRST_OPS` entry routes through `resolveContentsForJar`), so the single `reload` row is representative; the breadth is **unit-backed** by `scope.js`'s per-op wrapper coverage.
