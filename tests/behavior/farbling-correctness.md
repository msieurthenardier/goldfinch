# Behavior Test: Fingerprint farbling noises canvas and spoofs navigator

**Slug**: `farbling-correctness`
**Status**: draft
**Created**: 2026-06-05
**Last Run**: 2026-06-17-17-23-30 — [run log](./farbling-correctness/runs/2026-06-17-17-23-30.md) (**PASS** on the F9 eval-tool apparatus; promotion `draft→active` deferred to the next Electron upgrade per the note below, operator's call)

> **Draft — authored during Flight 4 (CI/CD hardening) planning as a Flight-3 carry-forward; apparatus migrated to the Goldfinch MCP `evaluate` tool in Flight 9 (DD5), not yet run on it.** Run and promote to `active` **before the next Electron major upgrade** — farbling is the largest privacy-unique feature with no unit or behavior coverage, and it rides on canvas/`navigator` prototype hooks installed at document-start that a Chromium bump could silently break. The formal Witnessed run + promotion is the Flight-9 verify-integration leg (leg 8).

## Intent

Verify that Goldfinch's anti-fingerprinting **farbling** actually fires on the live app: with Shields' `farble` active (the default), a page's `canvas` reads are perturbed with deterministic per-seed noise, and `navigator` fingerprint surfaces are spoofed to constant values that mask the real machine. This needs a behavior test rather than a unit test because the farbling is installed as **prototype patches inside the webview guest at document-start** via a synchronous `ipcRenderer.sendSync('shields-farble', …)` round-trip (`src/preload/webview-preload.js`) — only the running Electron app proves the hooks are installed, wired to the per-jar seed, and observed by real page JS. The hooks live in the **guest main world** (web webviews run `contextIsolation:false`), so the test measures them with the **Goldfinch MCP `evaluate` tool evaluating in that guest main world** — the same world the hooks patch; evaluating anywhere else would read the unmodified real values. The behavior also proves three properties a static check can't: noise is **applied** (not pass-through), **stable within a session** (a site re-reading the same canvas gets the same fake), and **seed-dependent** (a different jar / new identity changes the persona).

## Preconditions

- Goldfinch is running via **`GOLDFINCH_AUTOMATION_DEV_MINT=1 GOLDFINCH_AUTOMATION_ADMIN=1 npm run dev:automation`** — this binds the in-process MCP automation surface on the loopback transport and auto-mints the keys to stdout. The eval tool runs `webContents.executeJavaScript` in-process (zero CDP); this apparatus does **not** use any remote-debug / CDP port. **Both keys are needed**: the `default` **jar key** authorizes the core reads on that jar's own guest (Steps 2–5); the **admin key** is required for Step 6, because proving seed-dependence needs a **second seed** and the dev recipe mints only the one `default` jar key — reaching a second jar (two-container) or driving the chrome New-Identity control (Variant) both need admin. (If avoiding admin is wanted, a second jar key can be minted manually via the Settings → Automation UI; the two-container path then becomes pure-jar with two jar keys.)
- The keys are captured from the `AUTOMATION_DEV_MINT {"key":…,"adminKey":…}` stdout line printed at launch.
- An MCP client connects to **`127.0.0.1:$GOLDFINCH_MCP_PORT/mcp`** (default `49707`; pin the var once at launch and reuse it in every URL) with an **`Authorization: Bearer <key>`** header — the jar key for the core reads, the admin key for Step 6. A keyless request is a by-design `401`, so the readiness probe must attach the key. (`scripts/lib/mcp-client.mjs`'s `connectAutomation` is a ready helper if a script-side smoke is wanted; the formal run uses the behavior-test crew's registered MCP apparatus.)
- Shields' `farble` capability is **active** for the test site. It defaults to `true` (`src/main/shields.js:16`, `shields.active('farble', site)`); confirm Shields is not toggled off for the test origin before judging.
- A local HTTP page is served at a known `http://127.0.0.1:PORT/` URL (e.g. `python3 -m http.server` in a fixture dir). The page need only load over a legitimate scheme; the canvas/navigator reads are performed by the apparatus via `evaluate` in the guest main world (the prototype hooks patch all later-created canvases, so no special fixture markup is required). **Why HTTP, not file://**: the page must load over a normal scheme so farble activates for a real site origin — the drive tools also refuse non-`http(s)` URLs.
- **Active precondition probe** (Step 1): confirm the MCP surface answers an authenticated handshake (with `evaluate` present) and the fixture page is reachable before exercising farbling.

## Observables Required

- browser (guest main-world JS values — `navigator.hardwareConcurrency` / `navigator.deviceMemory`, `canvas.toDataURL()` / `getImageData` samples — measured via the **Goldfinch MCP `evaluate` tool** on the guest `<webview>` wcId; the eval tool runs `webContents.executeJavaScript` in the **guest main world** where the farbling hooks live, ZERO CDP, return value is JSON-serializable). **The apparatus must target the `<webview>` GUEST wcId — acquired via `enumerateTabs` and filtered to the jar's tab — NOT the top-level Goldfinch chrome target.** The farbling hooks are installed only in the guest's page context; evaluating in the chrome shell would read the unmodified real values.
- browser (New-Identity Variant only — the chrome "New Identity" control for the active jar, used to reroll the seed — measured via the Goldfinch MCP **admin** surface: `readAxTree` / `click`, or `evaluate(chromeWcId, 'newIdentity()')` on the chrome renderer global).
- shell (precondition probes: MCP surface answers an authenticated `initialize`/`tools/list`, fixture HTTP 200 — measured via the MCP client / Bash / curl).

## Steps

| # | Actions | Expected Results |
|---|---------|------------------|
| 1 | **Active-precondition probe.** Connect the MCP client to `http://127.0.0.1:$GOLDFINCH_MCP_PORT/mcp` with `Authorization: Bearer <jar-key>`; perform an `initialize` handshake and call `tools/list`. Also `curl` the fixture page URL. | The authenticated `initialize` succeeds (handshake ok) and `tools/list` includes the **`evaluate`** tool (assert presence — do NOT assert an exact tool count). The fixture returns HTTP 200. (A keyless probe returning `401` is by design — that is why the key is attached.) If either authenticated probe or the fixture fails, halt — preconditions not met. |
| 2 | In the running app, open a tab in the jar (via the `openTab` MCP tool with the jar key, or operator setup) and navigate it to the HTTP fixture page; wait for load. Confirm Shields is active (farble on) for this origin. Then call `enumerateTabs` and record the **guest `<webview>` wcId** for the fixture tab as **G** (filter to the jar's tab — this is the eval target, not the chrome). | The fixture page renders in the tab; Shields indicates `farble` is active (default). `enumerateTabs` lists the fixture tab and yields its guest wcId **G**. |
| 3 | **navigator spoof.** `evaluate(G, 'navigator.hardwareConcurrency')` and `evaluate(G, 'navigator.deviceMemory')`. | Both return exactly `8` — the spoofed constants (`webview-preload.js:325,330`), masking the real machine's core count / memory. (A real, un-farbled value would typically differ from 8 and vary by machine.) |
| 4 | **Canvas noise is applied + stable within session.** `evaluate(G, '<expr>')` where `<expr>` is a **single self-contained expression** that creates a canvas, draws a fixed image (e.g. text + a gradient rect), and returns `canvas.toDataURL()` — capture the returned data-URL string as **A**. Without reloading or changing identity, run the **identical** expression again and capture the returned string as **A2**. (If a one-expression draw is unwieldy, `injectScript(G, '<draw helper>')` immediately followed by `evaluate(G, '<call helper + toDataURL>')` — pair them with no intervening navigation.) | `A === A2` — the farbled output is **stable within the session** (a site re-reading the same canvas gets the same fake, per the deterministic per-`(seed,index)` hash, `webview-preload.js:241-246`). Both are JSON-serializable strings returned directly by `evaluate`. Record `A` for Step 6. |
| 5 | **Noise is non-trivial (sanity).** `evaluate(G, '<expr>')` that draws the same fixed image and returns a **small sampled subset** of `getImageData` bytes (e.g. a handful of pixels' RGBA, JSON-serializable — NOT the full array). Compare the sampled bytes against the flat/expected values a no-noise render of the same draw would produce. | The sampled pixels are perturbed — farbling is **applied**, not pass-through (the `getImageData` hook noises ~1/8 of pixels by ±1, `webview-preload.js:247-257`). (If a clean reference is impractical, treat the Step 6 cross-seed difference as the authoritative proof and mark this step a soft check.) |
| 6 | **Seed-dependence (two-container cross-seed — PRIMARY).** With the **admin key** (or a second jar key — both deliver a second seed), `openTab` the **identical** fixture URL in a **second, different jar** (a distinct session → a distinct per-jar seed via `seedForSession`, `main.js:797-808`); `enumerateTabs` → record that guest's wcId as **G2**. Run the **identical** canvas-draw expression from Step 4 in G2 and capture its `toDataURL()` as **B**. Also re-read `navigator.hardwareConcurrency`/`deviceMemory` in G2. | `B !== A` — the same fixed draw produces **different** canvas noise across the two jars, proving the noise is keyed to the per-jar seed (not a fixed transform). `navigator.hardwareConcurrency`/`deviceMemory` remain `8` in both jars (the spoof is a constant, not seed-derived). **Distinct-jar guard**: the two tabs MUST be in different jars — opening the fixture twice in the *same* jar yields the same seed and a false `B === A` negative. |
| 7 | **Control / farble-off (optional, if cheaply togglable).** Toggle Shields' `farble` off for the origin, reload the fixture (farble config is read synchronously at document-start, so a reload is required), then `evaluate(G, 'navigator.hardwareConcurrency')` and `evaluate(G, '<draw + toDataURL>')`. | With farble off, `navigator.hardwareConcurrency` returns the **real** machine value (not forced to 8) and the canvas read matches the un-noised reference — confirming farbling is what produced the Steps 3–6 differences, not some unrelated effect. (Skip if Shields-off-per-origin isn't readily drivable; note as untested.) |

## Out of Scope

- WebGL fingerprint farbling (`getParameter` vendor/renderer spoof, `webview-preload.js:298-308`) and AudioContext farbling (`webview-preload.js:310-320`) — extend with dedicated steps once the canvas/navigator core is promoted.
- The exact noise *distribution* / cryptographic quality of the per-seed hash — this spec proves applied + stable + seed-dependent, not statistical soundness.
- Tracker blocking and tracking-param stripping — covered by `core-browsing-shields`.
- Scheme-guard / hostile-URL behavior — covered by `tab-scheme-guard`.
- The eval tool's own contract (JSON-serializable return, internal-session exclusion, jar-scoping) — covered by the Flight-9 eval-tool unit tests and `mcp-drive-end-to-end`; this spec uses `evaluate` as the apparatus, it does not test it.

## Variants (optional)

- **New Identity reroll** (alternative to Step 6's two-container cross-seed): with the **admin key**, drive the chrome **New Identity** control for the active jar — `evaluate(chromeWcId, 'newIdentity()')` (or `readAxTree`/`click` the control), where `newIdentity()` (`renderer.js` — `newIdentity` function) calls `identity-new` (rerolls the per-session seed via `rerollSeed`, `main.js` — `rerollSeed`, from the `identity-new` handler `main.js` — `ipcMain.handle('identity-new', …)`) and **then reloads the guest** via `window.goldfinch.tabNavigate({ wcId: tab.wcId, verb: 'reload', args: [] })` (`renderer.js` — inside `newIdentity`). The reload is required because the guest captures `SEED` once at document-start (`webview-preload.js:233`), so a rerolled seed only takes effect on the next load. Wait for the reload to settle, re-acquire the guest wcId via `enumerateTabs`, then run the identical draw and read `toDataURL()` as `B`; assert `B !== A` (`navigator` stays `8`). This exercises the real "New Identity" UX, but the reload-timing window makes it flakier than the two-container path — which is why two-container is the primary Step 6. Use New Identity when the UX itself is what's under test.
