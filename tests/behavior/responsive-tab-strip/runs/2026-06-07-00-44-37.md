**Step-6 finding, resolved by the Orchestrator as a CDP-apparatus artifact (NOT a user-facing bug):** the Executor/Validator observed that a discrete CDP pointer *teleport* directly onto the `<webview>` (e.g. mouseMoved to 700,400) did NOT release the freeze (the webview captures that single event), whereas a teleport to host chrome did. Layout: `#tabstrip` bottom y=39, **`#toolbar` y=39–89**, `#webviews` top y=89. A follow-up **continuous (stepped) downward move** released the freeze the instant the pointer crossed into the toolbar (y=40 → tabs 138→158px), *before* reaching the webview. Since a real mouse moving from the strip into the page always traverses the intervening toolbar (host chrome), the freeze releases in normal use; only a discrete pointer warp that lands on the webview without crossing chrome (a CDP-only motion) leaves it frozen. No fix required. Optional future hardening (release on webview pointer-capture) is unnecessary given the toolbar always intervenes.
