### 2. Step 2's shape assertion was UNMEASURABLE — a row that could not fail

The row demanded the census carry "no own properties beyond indices (no
`incomplete` marker)". But `serialize()` is `JSON.stringify`, and **`JSON.stringify`
silently drops an array's non-index own properties**. The Validator verified it in
one line rather than taking it on report:

```
own keys before:  [ '0', '1', 'incomplete' ]
stringified:      [{"wcId":1},{"wcId":2}]
WIRE-IDENTICAL:   true
```

So over the wire a plain array and an array carrying `incomplete` are **the same
bytes**, and **no MCP client can ever fail this row**. The Executor's own
`wire_isPlainArray_postParse: true` was **true by construction** — it would report
`true` against a fully broken product. **A textbook latent false-pass.**

The decomposition matters: **array-ness vs wrapper IS measurable** (a
`{tabs, incomplete}` wrapper parses as an object). **"No own properties beyond
indices" is not** — and that is exactly the half DD1 cares about (the jar facade's
`.filter()` hazard). It is **main-side**, and properly pinned at
`test/unit/automation-tabs.test.js:721` with a genuine positive control
(`marked.incomplete = [2]` ⇒ `isPlainArray` false, plus an explicit assert that
`Array.isArray` alone cannot catch it).

**Folded**: the wire asserts array-ness + `windowId` + `jarId`; the unit pin is
cited for the rest; **and the reason is stated** (`JSON.stringify` drops non-index
own properties) — without it, a future author helpfully re-adds the assertion.

### 3. Step 7 acted on a background tab the spec never provisioned

Steps 1-4 placed exactly one tab in window 2, and it was **active** there — so at
row 7 no background tab existed. "Background" was also undefined: *not-active-
within-its-window* vs *in a non-last-focused window*. **DD6/S1's substance is the
latter.** The Executor provisioned a second tab as labeled **SETUP**, keeping it
out of the assertions and making the target background under **both** readings.

**Validator's ruling: row 7 stands** — the improvisation was a superset of what
the row needs and the observable is untouched by a second tab's existence. But it
cost two things: row 7's `activateTab` made its target active, **forcing row 8
onto a different tab** — so the pair varied *tab and op* where it should vary only
op; and it dodged the harder case (`activateTab` on a tab **already active** in a
non-focused window, where the only work left is the raise).

**Folded**: "background" defined; the second tab provisioned in setup; **rows 7
and 8 now act on the same tab**; an already-active variant added.

### 4. A REAL PRODUCT DEFECT — introduced by this flight, in its own blast radius

`mcp-tools.js` described **`readDom`** as "(foreground-first)". The shipped op does
the opposite — DD6 deleted its activate branch. **The Validator found a second the
Executor missed: `evaluate` carried the identical false claim.**
`captureScreenshot` and `readAxTree` say "foreground-first" **correctly** — they
still activate.

**Why it is a defect and not a nit**: the **operator-facing docs were already
correct** (`docs/mcp-automation.md` says readDom "Does NOT foreground its target").
So the human documentation was updated for DD6 and the **machine-readable tool
descriptions were not** — and the consumer that matters most here, an agentic MCP
client choosing how to call a tool, reads the stale one. A client could reasonably
pick `readDom` **as a raise primitive** and silently get no raise: **the exact
hazard DD6 set out to retire.** This is also the concrete instance of the gap leg 3
flagged ("nothing pins a tool description").

**Folded**: both descriptions fixed, and the description pin **extended to cover
them as a pair** — present on the raise half (its own same-run positive control),
absent on the no-raise half. **Mutation-controlled against the real artifact**:
baseline 86 pass → revert `readDom`'s text → **85 pass, 1 fail** → restore → 86 →
revert `evaluate`'s text → **85 pass, 1 fail** → restore → 86.
