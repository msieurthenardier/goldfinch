## Orchestrator Notes

**Finding 1 — coordinate-based `click` requires device-pixel coordinates (classify: LIKELY-REGRESSION / coordinate-space defect on the WebContentsView surface — FD triage).** Under WSLg `window.devicePixelRatio = 1.25`, `click(wcId, x, y)` lands at the target only when x/y are **device pixels** (CSS×DPR). CSS-pixel coordinates (the natural output of getBoundingClientRect / any layout read) miss. Reproduced deterministically: `click(252,412)` [CSS] → no hit; `click(315,515)` [device] → hit; 3/3 each way. This is a real driving-ergonomics defect for any DPR≠1 display. It is **NOT attributable to this flight's Leg-2 keyboard-bridge fix** — the keyboard primitives (`typeText`, `pressKey`) route correctly and need no scaling; the issue is mouse-coordinate mapping, most plausibly a WebContentsView `sendInputEvent` DIP-vs-device-pixel mismatch introduced by the migration. Compounding it: `readDom` and `readAxTree` (the spec's named coordinate sources) carry no geometry, so coordinates are only derivable via `evaluate`.

**Finding 2 — `captureWindow` is admin-only (classify: SPEC-DRIFT).** The jar identity is refused `captureWindow`; Step 9's parenthetical wrongly lists it in the jar-driven observe set. The cross-check was completed with the admin key. Recommend the spec move `captureWindow` to the admin-only column alongside `getChromeTarget`.

All other steps met their observable checkpoints on the jar identity as written.
