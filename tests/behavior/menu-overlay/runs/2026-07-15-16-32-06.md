## The band-1 ruling (recorded because the reasoning matters)

The undeclared ▾ band is an **undeclared-but-benign mutable region — not a
regression, and not sheet residue**. It is CSS `:hover` on the trigger the
synthetic pointer is still resting on after step 5's injected
`click(1, 538, 23)`. The Validator's evidence is quantitative and convergent:
the tint is a pure uniform multiplicative darkening with measured per-pixel
ratio **0.8788 ± 0.0096** → alpha **0.1212**, matching
`src/renderer/styles.css:182-184` (`#newtab-pill .icon-btn:hover { background:
rgba(0,0,0,0.12) }`) to three decimals; it is confined exactly to the button
rect; nothing renders below it (y40-300 x500-790 maxdelta 0 once the clock
glyph's right edge is excluded); and `aria-expanded === "false"`.

It cannot be an F7 artifact: the hover rule is commit `dc0604f8` (2026-06-06),
present identically on `main`, and leg 1's working tree touches only
`src/main/main.js`, `src/main/window-registry.js`, and the new
`find-overlay-manager.js` — **no renderer files at all**.

**But the Executor's stated reasoning was wrong, and the Validator caught it.**
The Executor argued the band "would reproduce identically pre-F7 (step 5 clicked
that same trigger on the 2026-07-06 6/6 run)". The prior run log contradicts
this twice: its checkpoint 2 records the trigger opened **via `evaluate`** — a
DOM-level `.click()` that never moves the pointer and so never sets `:hover` —
and its checkpoint 6 explicitly records "only deltas are clock, ticks, and the
automation-activity badge". The band **is** new relative to the last run; its
cause is simply *this run's driver choice* (injected `click` via
`sendInputEvent`), not the conversion. Right answer, wrong reason — and the
wrong reason is the kind that would license waving through a real delta next
time. **The pass rests on the guest region being bit-exact and the tint being a
measured 0.12 hover composite, not on the Executor's counterfactual.**

## Spec-quality observations (recommended; nothing was edited)

1. **Find-overlay typing target — erratum recommended.** The spec pins the
   multi-view rule for the sheet but not the find bar. `typeText(guestWcId)`
   silently types into the guest; the find overlay is a separate view requiring
   its own wcId. This failed **silently** and was caught only on pixels — exactly
   the failure mode rendered-state precedence exists to catch, which argues for
   pinning it rather than relying on the next Executor's luck. Extend the
   Observables apparatus-nuance note to cover the find overlay symmetrically.
2. **Declare trigger hover/focus state as an expected mutable region, or pin the
   driver.** Step 6's declared regions assume the pointer parks nowhere
   meaningful. Any run driving a menu by injected click leaves `:hover` on the
   last trigger and reproduces exactly this undeclared band. Options: declare it
   mutable; pin the driver to `evaluate`-based clicks (what the prior run did);
   or park the pointer before the final grab. Left undeclared it recurs on every
   injected-click run and burns Validator time re-litigating it — as it did here.
3. **Step 6's Intent and Expected Result are not aligned.** Intent scopes
   return-to-baseline to "**the guest region** returns pixel-equivalent to
   baseline"; the Expected Result says "**Frame** pixel-equivalent". Chrome-side
   deltas satisfy the first and violate the second — that gap is what made this
   finding ambiguous enough that the Executor declined to rule. Tighten the
   Expected Result to the guest region, with the enumerated clauses (no menu, no
   residue/strip, no find bar, guest full-height) carrying the chrome-side
   assertions.
