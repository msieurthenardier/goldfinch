## THE CENTRAL FINDING: this PASS does not prove what the run was commissioned to prove

The FD selected this spec for the triple on the ground that it exercises the
window-`blur` conditioning leg 1 deleted. **That rationale is wrong**, caught by
the Executor and confirmed by the Validator against the spec text. The spec
scopes blur **out**, twice and explicitly — `:97-100` (*"Scripted focus can't
fake OS blur … HAT-scoped"*) and `:144-145` (Out of Scope: *"OS/app-switch blur
dismissal … HAT-scoped"*). Zero blurs were driven; `win.on('blur')` never fired.
**The spec was selected on its topic name; its text disclaims the mechanism the
ruling wanted covered.**

**The Validator went further, and this is the part that matters.** Compare the
code directly. HEAD (`b607411:main.js:1234`):

```js
// DD7 blur conditioning (M09 F6 Leg 4): fires ONLY when the blurred window IS
// the sheet's current attachment window — else opening a menu in window B is
// killed by A's in-flight blur (the two-window open handoff).
win.on('blur', () => {
  if (menuOverlay.getAttachedWindow() === win) menuOverlay.closeMenuOverlay('blur');
});
```

Working tree (`main.js:1171`): `win.on('blur', () => { sheet.closeMenuOverlay('blur'); });`

**In a single-window rig these are behaviorally identical.** Menu open in the
only window ⇒ `getAttachedWindow() === win` is true ⇒ both close. No menu open ⇒
the old guard fails, the new `closeMenuOverlay` is idempotent and no-ops ⇒ same
observable. There is no third case with one window. So **no single-window test
can distinguish the pre- and post-deletion code — even one that drove a real
blur on a platform that delivered it.** The deleted guard's own comment names
the only scenario that exercises it: *"opening a menu in window B is killed by
A's in-flight blur (the two-window open handoff)"* — two windows and a real OS
blur.

**Disposition**: the spec keeps its place in the triple on the honest ground —
**single-window invariant coverage** (it drives the per-window `sheet` slot, the
`closeMenuOverlay` family, model-replace, geometry, and the DD5 Escape→refocus
through the converted wiring). The triple proves what it was built to prove and
does not prove what its selection note claimed. **The DD7 blur-conditioning
deletion remains genuinely unverified and is unreachable by any single-window
test on structural grounds.** Carries are recorded in the flight log.

## The step-2 instrument chain (the run's most transferable finding)

Step 2's Expected Result is a three-way conjunction whose third conjunct is *"no
focus is stolen to the trigger"*. **The spec names no observable for it.** What
followed is a three-instrument false-reading chain that nearly shipped wrong in
*both* directions:

1. **Instrument 1 — a broken AX parser gave a false PASS.** The parser looked
   for a top-level `focused` field; it actually lives inside each node's
   `properties` array (and `RootWebArea` reports `focused=true`, a second trap).
   It returned `found:false`, which *looks exactly like* "focus was not stolen".
   The conjunct was **confirmed by malfunction, not by measurement**. The
   Executor later root-caused this same parser bug on step 6 — where it produced
   a false FAIL — and re-ran that step, but step 2 was never revisited, so the
   false pass survived.
2. **Instrument 2 — pixels cannot discriminate.** The Validator caught the
   tainted conjunct, tried to settle it independently, and could not: it cropped
   the kebab rect from the step-2 capture and compared against step 6, where
   focus is *proven* on the kebab — **indistinguishable, no focus ring in
   either**. It returned INCONCLUSIVE rather than a charitable pass.
3. **Instrument 3 — the AX tree cannot discriminate either, and the FD's fix
   instruction was itself wrong.** The FD said re-measure *"with the fixed
   parser"*. Following that literally would have produced a **false FAIL — the
   mirror image of the original false pass** — because AX `focused` tracks the
   chrome document's `activeElement` and **persists even when that webContents
   holds no OS focus**. It reads *identically* (`More`/#kebab) for the Escape
   control (focus genuinely restored) and for T2 (focus not restored). **The
   Executor added an Escape positive control on its own initiative**, which is
   the only reason the third reading is a measurement of absence rather than a
   third artifact.

**The re-measurement** (cold relaunch, fresh scratch profile, boot bracket clean,
actions identical):

| timepoint | AX focused node | chrome activeElement | **chrome.hasFocus()** | sheet.hasFocus() |
|---|---|---|---|---|
| control: after Escape | `More` (#kebab) | `BUTTON#kebab` | **true** | false |
| T0 anchor (omnibox) | `Address and search bar` | `INPUT#address` | true | false |
| T1 menu OPEN | `More` (#kebab) | `BUTTON#kebab` | **false** | true |
| T2 after outside-click | `More` (#kebab) | `BUTTON#kebab` | **false** | false |

**No focus stolen.** At T2 **no webContents holds focus at all**, so the trigger
cannot be holding it. The `#kebab` `activeElement` at T2 is **residual from T1's
opening trigger click** — the T0 omnibox anchor establishes that cleanly
(`INPUT#address` *before* the opening click, `BUTTON#kebab` after, unchanged
through dismissal), so `#kebab` demonstrably predates the dismissal. This matches
`menu-overlay-manager.js:310` — `if (reason === 'escape' || reason ===
'activated') focusChrome(...)`, with the comment at `:307-309` stating *"Every
other reason moves NO focus"* — but the Executor **measured first and used
source only as corroboration, which is the correct order**.

**Why the control is sound, per the Validator — stronger than the Executor
claimed**: a single control at the head of a run only proves the instrument
worked *before* the measurement. What makes T2 meaningful is that **T0 supplies
a second `true` reading inside the sequence**, and both observables then move in
**both directions** across T0→T1→T2 — so neither is stuck. The instrument is
demonstrably live *adjacent to* the measurement, not just at boot.

**Validator's precision on the discriminator**: `hasFocus()` is the **missing
necessary** discriminator, not the whole observable. *"Focus is on X"* is itself
a conjunction — `chrome.hasFocus() === true` **AND** `activeElement === X`.
Neither alone decides it (`hasFocus()` alone can't say *which* element — focus
restored to the *omnibox* reads `true` and still violates the conjunct;
`activeElement` alone ignores whether anyone holds focus). At T2 it's moot
(`hasFocus()=false` on both kills the conjunct outright), but the distinction
governs how the spec should be worded.

## Spec-quality observations (recommendations only; the spec was never modified)

1. **Name the observable for step 2's focus conjunct — as a conjunction.**
   Specify *"focus is on X"* as `document.hasFocus() === true` **AND**
   `activeElement === X`. Naming `hasFocus()` as "the primary discriminator"
   alone invites the mirror-image error later.
2. **Demote the AX focused node GLOBALLY — the bigger fix.** The spec currently
   makes AX **primary** and `hasFocus()` a corroborator, in the Observables
   section (`:106-109`) and again in step 6's row. **The re-measure inverts that
   hierarchy**: AX `focused` is *structurally incapable* of deciding
   focus-vs-no-focus. Patching only step 2 leaves misleading guidance standing
   everywhere else. Add the `RootWebArea focused=true` false-positive too, which
   the spec also omits. (Step 6's **verdict stands** — it read the correct
   conjunction and was right *by construction*, but right **despite** the spec's
   ordering, not because of it.)
3. **Mandate the positive control for focus, as the spec already does for
   pixels.** The root defect is an **asymmetry**: the spec rigorously requires
   presence-before-absence for pixels (the sheet-compositing litmus) but requires
   nothing equivalent for focus — and that asymmetry is exactly where a broken
   instrument survived a full pass. Generalize: **any absence assertion needs a
   same-run demonstration that the instrument can read presence.**
4. **Generalize "re-locate before each click — do NOT cache" (`:76-79`) to a
   coordinate-MEASUREMENT rule** covering pixel probes, not just clicks. The
   spec pre-warned about exactly the drift that bit step 9 (the ▾ trigger shifts
   right as tabs are added: x=282 at 1 tab → x=526 at 2), and the Executor
   honored it for **clicks** but not for its **pixel probe**.
