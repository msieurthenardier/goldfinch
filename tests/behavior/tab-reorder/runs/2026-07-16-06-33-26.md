## Summary

**Why this run exists**: `tab-reorder` is **the F2 regression net**. M09 F8 leg 3 changed the drag's
arm threshold from `Math.abs(dx)` to **`Math.hypot(dx, dy)`** and interposed a **zone model**
(`tab-drag-zone.js`) that now runs on **every `pointermove`**. This spec is what stands between that
change and a silent reorder regression.

**Verdict: 8/9 PASS. The reorder contract survives.** Step 3 proves the reorder half survived the
interposition — model-predicted drop index (**recomputed independently by the Validator** from the
recorded rects), DOM / a11y / rendered agreement, drag-start activation, cleared transforms, and the
`reorder` zone held (`detachingSeen: false`). Steps 5–9 cover keyboard reorder, the DD1 consumer, and
the DD2 click model. **If F8 had broken in-strip reorder, this run would have caught it.**

**Step 4 is INCONCLUSIVE by construction, and that is what forces `partial`.**

## Step 4 — INCONCLUSIVE, and it is F8's central finding landing on a sibling spec

**The Validator pre-committed this verdict before the row ran, and the rig returned exactly the
predicted numbers:**

```
screenX: 564   screenY: 260   outerWidth: 1432   outerHeight: 942
```

Step 4 asserts the window did not move, reading `window.screenX`. Its WSLg escape hatch fires only
*"if the apparatus reports a **constant placeholder (e.g. always 0)**."*

- **`564` is not zero — so the hatch does NOT fire, and a literal reading scores the row a hard PASS.**
- **But `564` is the exact value F8's transport spike proved is a CACHED FICTION**, against an
  independent Win32 `GetWindowRect`/RAIL witness: *after a **real** OS move, Electron `getBounds`
  still read `{x:580,y:270}` and renderer `window.screenX` still read **564**. Electron never
  noticed. A real OS move fires **no Electron event at all**.*
- ⇒ **"the window moved" and "the window didn't move" read IDENTICALLY. Discrimination zero.**
- The spec offers `outerWidth`/`outerHeight` as a backup cross-check. **It carries no power either**:
  a *move* does not change outer size, so it cannot detect the thing Step 4 exists to detect.

> **The escape hatch is calibrated to the wrong tell.** It tests for **zero** when the actual failure
> mode is **frozen-but-plausible**. This is F8's own thesis, verbatim, arriving in a spec written
> before it: **`{0,0}` is obviously wrong on sight; `564` is not.** A hatch that fires only on the
> loud failure cannot catch the quiet one.
>
> **The verdict is INCONCLUSIVE, not PASS — and byte-identical values are precisely what a dead
> instrument reports.** This is a **spec defect for the debrief, not a product finding.** The only
> known discriminating instrument is the Win32 `GetWindowRect`/RAIL witness the spike used.

## THE OWNERSHIP GAP — `Math.hypot` is owned by NOBODY, and two specs each believed the other held it

**This is the run's most important finding and it was found by reading, not by failing.**

`tab-tearoff.md` stated: *"`tab-reorder.md` owns the F2 drag contract, **the `Math.hypot` arm
threshold**, and the keyboard reorder."* **It does not.**

| Candidate owner | Why it does not cover the threshold |
|---|---|
| **`tab-reorder`** (this spec) | Its only drag holds **y constant ⇒ `dy = 0` ⇒ `Math.hypot(dx, 0) ≡ Math.abs(dx)`**. It **cannot distinguish the new threshold from the old.** |
| **`tab-tearoff`** | Rows 3/4 **deliberately share a common `DROP_X`**, so **both carry `dx ≠ 0` and arm under `abs` too.** The very device that makes them a clean control blinds them to the threshold. |
| **The unit suite** | **No test references `hypot` or `DRAG_ARM_THRESHOLD_PX`.** `tab-drag-invariants.test.js` pins call-site arity and layout-neutrality, **not threshold behavior**. |

**`renderer.js` names the reason for the change: a straight-DOWN gesture (`dx = 0`, `dy > 5`) must
arm, and under `abs(dx)` it never would.** ⇒ **The exact case the change was made for is unfalsified
by the entire suite.**

**Owed**: a **unit** test over the threshold predicate (`dx=0, dy=6` arms; `dx=0, dy=4` does not) —
**not** another live drag. `tab-tearoff.md`'s false ownership claim has been **corrected** so the gap
is recorded rather than re-inherited.
