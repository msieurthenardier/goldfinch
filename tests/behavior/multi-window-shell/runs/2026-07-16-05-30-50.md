## Erratum 2, folded a SECOND time — **it was folded by NAMING and never by MEASURING**

**This is what forces `partial`, and it sits in the exact clause the last fold touched.**

V1's corollary read: *"`captureScreenshot(<window 1's sheetWcId>)` and `captureScreenshot(<window 2's
sheetWcId>)` each render **their own** open menu."* Measured this run:

| Reading | Result |
|---|---|
| `captureScreenshot(<W1 sheetWcId 5>)` vs `captureScreenshot(<W2 sheetWcId 6>)` | **BYTE-IDENTICAL** — `md5 d9e34c95…`, 15610 bytes each |
| `captureScreenshot(<sheet wcId>)` with the sheet **HIDDEN** | `hasImage: true` — **captures fine** |

- **Byte-identity is not a bug** — both windows' kebabs render **the same six items at the same
  geometry**, so two distinct `WebContentsView`s *necessarily* produce identical pixels.
- **But it means the clause cannot attribute a capture to a window** — byte-identical output is
  exactly what capturing the **same** sheet twice would produce. **It cannot discriminate two sheets
  from one, which is the property V1 exists to prove.**
- **And a hidden sheet captures fine**, so the clause **would return an image with no menu open at
  all.** **It could not fail.**

> **The 2026-07-15 fold moved this clause from vacuity-by-ABSENCE (the chrome wcIds cannot contain the
> menu — it showed nothing) to vacuity-by-INDISTINGUISHABILITY. It satisfied its letter — "name the
> sheet wcIds" — and missed its purpose: measure something that can fail.**
>
> **That is the SAME meta-pattern as erratum 1**, which was added *specifically to make a failure
> loud* and named an op that could not fail. **Third instance in this spec of an instrument whose
> TARGET was repaired while its DISCRIMINATION was never examined.**

**Why this forces `partial` rather than a footnote** (the Validator's reasoning, adopted): the bar was
explicit — *if any instrument in this spec still discriminates nothing, that is a finding, not a
pass.* The step-8 `index 1` residue is **inert prose**. **This is a live ACTION clause** —
*"Corroborate on pixels: `captureScreenshot(…)`"* — instructing a measurement that cannot fail, **and
`captureScreenshot` raises and activates**, so the spec orders a **state-perturbing read on state the
next row (V2) depends on** — contradicting this rig's own documented hazard. **This spec's own
