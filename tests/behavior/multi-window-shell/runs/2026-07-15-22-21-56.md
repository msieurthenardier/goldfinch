## Erratum 1 (LOAD-BEARING) — the gate leg 4 added names an op that cannot perform it

Step 8's **marquee rewrite addition** — *"read `getHistory(R2)` and assert its
ENTRY COUNT … exactly 2 entries at index 1 with `canGoBack` true"* — and Step 2's
`getHistory(T2)` clauses are written against a **per-tab navigation-history op
that does not exist**.

The real op is **M08 F5's jar-confined visits reader**:
`getHistory(jarId: string, {query,limit,before})` → `{jarId, visits}`. Measured
live:

| Call | Result |
|---|---|
| `getHistory({wcId:12})` — spec-literal | **refused**: `bad-args — jarId required` |
| `getHistory({jarId:12})` — numeric | **refused**: `bad-args — jarId required` |
| `getHistory({jarId:'work'})` — presence demo | **works** — jar-wide visit log, 8 visits across *all* work tabs; **no** count/index/`canGoBack`, **no** tab scoping |

**`index` and `canGoBack` are unobservable on this surface by any op** —
`canGoBack` exists only as an internal guard inside `nav.js`'s `goBack`.

**The sharp point**: the gate exists to convert a silent page-mismatch into a
**loud count-mismatch**. As written it instead produces a **`bad-args` refusal on
every run, independent of whether `ERR_ABORTED` occurred** — so it
**discriminates nothing**. That is the precise defect class this flight has been
paying for all along, and it landed **in the assertion added to prevent it**.

**The behavior the gate protects is correct**: `evaluate(R2,"history.length")` =
**2**, and `goBack(R2)` → page 2 with both markers. **A spec-instrument defect,
not a product regression.** The Executor **did not substitute and green it** — it
measured the real observable, reported, and left the spec untouched.

**Folded**: both gates now read `evaluate(wcId, "history.length")`;
`index`/`canGoBack` dropped as unobservable; the reason recorded inline so the
instrument isn't helpfully restored.

## Erratum 2 (minor) — V1's pixel clause names the wrong wcIds

V1 said `captureScreenshot(C1)`/`(C2)` "each render **their own** open menu".
Measured: **`captureScreenshot(C1)` renders window 1's chrome with no menu.** The
menu renders only in the **sheet's** wcId.

This **follows necessarily from what the row itself asserts**: a per-window sheet
is its own `WebContentsView` with its own wcId, so the chrome wcId's capture
cannot contain it. The row's **primary** observable passed cleanly, so V1's
verdict doesn't hinge on it.

**Folded**: the clause now names the sheet wcIds (still per-wcId, so DD9's
no-`captureWindow` constraint holds).
