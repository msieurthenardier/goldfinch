## Executive Summary

Goldfinch is a clean, well-commented v0.2.2 codebase with sound core Electron hardening (webviews run `nodeIntegration:false`, the page main world receives no Node primitives, the chrome window uses `contextIsolation:true`, sensitive permissions are denied by default, `setWindowOpenHandler` denies native popups, CSP present, consistent `escapeHtml`, no secrets, correct `.gitignore`). The realistic threat model is a **hostile web page loaded in a `<webview>`**, and findings are weighted by whether a page can actually reach them. There are **no Critical findings**, but **seven Action Required** items: one directly hostile-page-reachable security gap (`open-tab` scheme bypass), a nine-major Electron version gap that ships Chromium CVEs, zero automated tests over security-critical privacy logic, an inaccurate public README, missing CI least-privilege token scoping, and a keyboard/screen-reader-inoperable tab strip. The codebase is **not flight-ready** for new feature work until the security and dependency-currency items are addressed.

## Findings by Category

### Category 1: Security

| # | Finding | Severity | New/Known | Recommendation |
|---|---------|----------|-----------|----------------|
| F1 | `open-tab` forwards a hostile page's `window.open()` URL to a webview `src` with no scheme filter | Action Required | New | Allowlist `http:`/`https:`/`about:blank` in `setWindowOpenHandler` and/or `createTab`; reject `file:`/`data:`/`javascript:` |
| F3 | `open-external` IPC passes any URL to `shell.openExternal` with no scheme allowlist (currently unused/latent) | Advisory | New | Add scheme allowlist before `openExternal`, or remove the binding while unused |
| F4 | `download-media` `saveDir` argument not containment-validated | Advisory | New | Defense-in-depth: confirm `saveDir` is a dialog-approved directory (it currently always is — see Details) |
| F5 | Download filename: `uniquePath` strips separators but not `..`/leading-dot/Windows-reserved names | Advisory | New | Strip leading dots, reject reserved names, assert `path.resolve` containment |
| F6 | CSS injection via unescaped video `poster` in `backgroundImage` | Advisory | New | Sanitize via `CSS.escape` or restrict `poster` scheme to `http(s)/data/blob` |
| F7 | `containers.json` parsed without shape validation | Advisory | New | Validate `partition` prefix, dedupe ids, coerce name/color to safe strings |

**Details:**
- **F1** (`src/main/main.js:58-61` → `src/preload/chrome-preload.js:36` → `src/renderer/renderer.js:1083,111`): the page-supplied `window.open()` URL flows verbatim through `open-tab` to `webview.setAttribute('src', url)` with no validation at any hop. A hostile page can `window.open('file:///etc/passwd')`, `data:`, or `javascript:`. `allowpopups` is set and the webview carries the goldfinch preload, widening impact. **This is the one finding a hostile page reaches directly** — highest-priority security item.
- **F4**: `saveDir` originates only from the native OS folder picker (`chooseDownloadDir()` → `dialog.showOpenDialog`, `src/main/main.js:101-107`), never from page content; a hostile page cannot supply it. The risk is purely second-order ("if the chrome renderer is compromised"), outside the threat model — hence Advisory, not High as initially estimated.
- **F5** (`src/main/main.js:88-99`): separators are stripped (so `../` collapses), leaving residual leading-dot/reserved-name risk within the user-chosen directory.
- **F6** (`src/renderer/renderer.js:355`; source `src/preload/webview-preload.js:113`): `poster` passes through `new URL()` which mitigates classic breakout, but it is the one sink not run through `escapeHtml`; fix for consistency under the chrome's `style-src 'unsafe-inline'`.
- **F7** (`src/main/jars.js:21-30`): tampered `containers.json` (requires local file write, so out of the web threat model) could collapse two jars onto one session, breaking isolation.

### Category 3: Dependency Health

| # | Finding | Severity | New/Known | Recommendation |
|---|---------|----------|-----------|----------------|
| F2 | Electron is nine majors behind (33.4.11 → 42.3.3), shipping Chromium/Electron CVEs reachable by hostile content | Action Required | New | Upgrade Electron deliberately (breaking, nine majors — own flight with a regression pass) |
| F9 | No `engines` field despite Node-version-sensitive toolchain (CI Node 20, dev Node 22) | Advisory | New | Add `engines.node` aligned with CI (e.g. `>=20`) |

**Details:** `npm outdated` confirms `electron 33.4.11 → 42.3.3` (nine majors); `electron-builder 25.1.8 → 26.8.1`. `npm audit` reports 10 highs, but the headline is misleading: **nine are the `electron-builder` dev toolchain (tar/cacache/node-gyp/app-builder-lib — build-time only, not shipped); only one is `electron` itself**, which is runtime-shipped and carries advisories (iframe-permission origin confusion, custom-protocol header injection, service-worker IPC spoofing) reachable by hostile web content. **Do not run `npm audit fix --force`** — it would churn the dev toolchain; the real remediation is the targeted Electron major upgrade (F2).

## Regression-Gate Recommendations

Two behaviors are worth pinning against future regression with **behavior tests** (real-environment, AI-driven; see `.claude/skills/behavior-test/AUTHORING.md` on the mission-control side). Author these inline during the relevant flight's design:
- **Scheme-bypass guard (F1)**: a spec that drives a page calling `window.open('file:///...')` and asserts the URL is rejected / not loaded into a webview. Hostile-page behavior doesn't fit a unit test.
- **Tab keyboard operability (F22)**: a spec that Tab-focuses the tab strip, switches and closes tabs via keyboard, and asserts ARIA state — verifying real keyboard/AT behavior the unit layer can't.

## Recommendations

1. **Close the one hostile-page-reachable hole first (F1)** — add the `open-tab`/`createTab` scheme allowlist; it's small and directly in the threat model.
2. **Plan the Electron major upgrade (F2)** as its own flight with a regression pass — it's the highest-leverage runtime security item; do **not** `npm audit fix --force`.
3. **Stand up a test + quality floor (F8, F10, F11)** so the privacy core has a regression net before further feature work, and fix the public README (F12).
4. CI least-privilege (F17) and the accessibility keyboard baseline (F22/F23) round out the Action Required set.
